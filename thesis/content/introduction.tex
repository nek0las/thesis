\chapter{Introduction}
\label{chapter:introduction}
\section{Motivation}
Modern computer architectures run programs concurrently in shared memory. Under this model, subtle behaviors can lead to bugs and security issues; for instance, \citet{10.1145/2676726.2676995} show that many common compiler optimisations are invalid under the C11 memory model. To study such behaviours systematically, \citet{10.1145/2627752} introduced the diy7 toolchain and the Cat specification language for describing weak memory models. Cat represents an execution as a graph built from binary relations, which enables concise axiomatic specifications and supports formal reasoning. Using Cat-style relations, \citet{10.1145/3458926ArmedCat} developed and maintained an axiomatic formal model for Arm, and \citet{10.1145/3140587.3062352Reparing} proposed a repaired account of sequential consistency for C/C++11. However, mechanization and proof construction are orthogonal problems, and only a limited number of solutions address both simultaneously.

Writing a proof for each architecture and each memory model on paper is time-consuming. Proof assistants can reduce boilerplate work by reusing existing proofs, and they also provide tactics to automate certain proof steps. In particular, \citet{10.1145/3110278Lean4Meta} present a metaprogramming framework for Lean, and Lean 4's expressive type system is well-suited to encoding the types and semantics of the Cat language. Many existing mechanizations focus only on the essential ideas and do not directly reuse Cat specifications. However, this approach introduces additional work because many memory models are already defined in Cat.

The Arm team mechanized the ARMv8 memory model in RCoq \citet{10.1145/3458926ArmedCat}, but this work targets only Arm and does not provide a general framework. Similarly, in his PhD thesis, Dennis develops an Agda framework for verifying the correctness of transformations between architectures without relying on Cat files \citet{DennisPhdThesis}.

In this project, we propose a domain-specific proof framework, LeanCat, to mechanize weak memory models in Lean 4. LeanCat relies on existing Cat files: it parses Cat specifications and introduces their semantics into Lean. It also provides necessary primitives and abstractions to help users prove interesting properties, such as whether one memory model is weaker than another. LeanCat greatly reduces the mechanization effort and the amount of boilerplate code required for weak memory transformation proofs.

\section{Contributions}

We design and implement LeanCat, a domain-specific proof framework that mechanizes weak memory models in Lean 4. LeanCat parses existing Cat specifications directly and introduces their semantics into Lean 4 using its metaprogramming framework, so that users do not need to translate or rewrite memory model definitions manually. We implement a core subset of the Cat language targeting LKMM, including support for tagged events, relation operators, and the acyclicity, irreflexivity, and emptiness judgments.

We define the core types and well-formedness conditions needed to represent candidate executions in Lean 4, and we develop a library of theorems about binary relations and acyclicity that bridge the gap between Lean 4's existing Mathlib library and the specific needs of weak memory model reasoning. In particular, we prove an acyclicity monotonicity lemma that serves as the central proof technique for comparing memory models.

We demonstrate LeanCat through two case studies. First, we check concrete litmus test executions against memory models, including SC, TSO, MIPS, and LKMM. Second, we formally compare memory models with each other. In particular, we present a machine-checked proof that Sequential Consistency is strictly stronger than Total Store Order, and Total Store Order is strictly stronger than MIPS, reducing the comparison to a monotonicity argument over acyclicity constraints. We also provide a counterexample execution that TSO permits but SC forbids, completing the strict separation.

We integrate execution graph visualization into the Lean 4 development environment using ProofWidgets, allowing users to inspect candidate executions graphically during proof development.

\section{Structrure of this thesis}

The remainder of this thesis is structured as follows.

Chapter~\ref{chapter:Background} introduces the necessary background. We begin with an 
overview of weak memory models, covering both operational and axiomatic perspectives, 
and walk through concrete examples using the Store Buffering litmus test. We then 
introduce the axiomatic memory models in detail, including Sequential Consistency, 
Total Store Order, and the Linux Kernel Memory Model. We follow this with the 
mathematical background on binary relations needed throughout the thesis, and conclude 
the chapter with an introduction to Lean~4 and its metaprogramming framework.

Chapter~\ref{chapter:design} describes the implementation of LeanCat. We present 
the core types used to represent memory models in Lean~4, including events, candidate 
executions, and well-formedness conditions. We then explain how we connect the Cat 
specification language to Lean~4 using its metaprogramming framework, describe the 
supported subset of Cat targeting LKMM, and discuss the key design choices and 
tradeoffs made during implementation.

Chapter~\ref{chapter:results} presents the results. We demonstrate LeanCat through two 
case studies: checking concrete litmus test executions against memory models, and 
formally comparing memory models with each other. In particular, we present a 
machine-checked proof that Sequential Consistency is strictly stronger than Total 
Store Order.

Chapter~\ref{chapter:discussion} discusses the limitations of our approach, including 
constraints imposed by Lean~4 itself and the subset of Cat that remains outside the 
scope of our current formalization.

Chapter~\ref{chapter:conclusion} concludes the thesis and summarizes our contributions.

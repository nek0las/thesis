\section{Visualization}
\label{sec:visualization}

In Lean 4, we use the \emph{InfoView} panel on the right-hand side of Visual Studio Code. We typically rely on this panel (via the Lean VSCode extension) to inspect the current goals and track the progress of a proof.

In our development, we use type classes together with \texttt{Set} and \texttt{RelSet} to define candidate executions. However, this representation can be difficult to reason about from a human perspective. Essentially, graphs in Lean 4 are encoded in a way that is convenient for machine processing, but not for human intuition. It would be much easier to understand the structure—especially to identify cycles—if we could directly visualize the graph.

To achieve this visualization, we use the tool ProofWidgets~\cite{nawrocki2023proofwidgets}\footnote{\url{https://github.com/leanprover-community/ProofWidgets4}}. Broadly speaking, it provides APIs that allow us to visualize data and proofs interactively. Let us look at an example of how it works in LeanCat.

The following program represents the X86 \emph{Store Buffering (SB)} litmus test (from Figure~\ref{fig:litmus-test}):

\begin{verbatim}
/-! ### X86 SB (Store Buffering) Litmus Test -/

private def sbEvents : Array Event :=
  #[initWx, initWy, inst1writeX, inst2readY, inst3writeY, inst4readX]

private def sbRf : Array (Event × Event) :=
  #[(initWy, inst2readY), (initWx, inst4readX)]

private def sbCo : Array (Event × Event) :=
  #[(initWx, inst1writeX), (initWy, inst3writeY)]

/-- Concrete SB litmus-test execution, built from the events and relations
defined in `Litmus.lean`. -/
def sbExecution : ConcreteExecution := {
  events     := sbEvents
  locName    := fun n => n
  threadName := fun n => n
  po         := computeDirectPo sbEvents
  rf         := sbRf
  co         := sbCo
  fr         := computeFr sbRf sbCo
  rmw        := #[]
}
\end{verbatim}

After defining the base relations (\texttt{rf}, \texttt{co}, and derived \texttt{fr}), we construct a candidate execution. The \texttt{co} relation can be derived directly from the litmus test, and \texttt{fr} is computed from \texttt{rf} and \texttt{co}.

Once the candidate execution is defined, we can use the command:

\begin{verbatim}
#html toOrderedHtml sbExecution
\end{verbatim}

This produces a clear graphical representation of the execution. From this graph, we can easily observe the presence of a cycle (W(x) = 1 -> R(y) -> W(y) = 1 -> R(x)), which indicates that the execution is disallowed under the memory model. This visualization can then serve as a useful starting point for proving the result formally.

\section{Library Theorems}

To support the case studies, we proved a collection of library theorems about binary relations,
acyclicity, and the primitive relations of candidate executions. These theorems bridge the gap
between Lean~4's existing \texttt{Mathlib} library and the specific needs of weak memory
model reasoning. We highlight the two most important results in detail and briefly summarise
the remaining supporting lemmas.

\paragraph{Acyclicity Monotonicity.}
The central proof technique for comparing memory models is \emph{monotonicity}: if a
relation~$r$ is acyclic and~$s \subseteq r$, then~$s$ is also acyclic. We formalize this as
\texttt{acyclicMono}:

\begin{lstlisting}
lemma acyclicMono
  {r₁ r₂ : SetRel Event Event}
  (hacyc : SetRel.Acyclic r₂)
  (hsub : ∀ a b, (a, b) ∈ r₁ → (a, b) ∈ r₂)
  : SetRel.Acyclic r₁
\end{lstlisting}

The proof proceeds by lifting the edge-wise inclusion to a transitive-closure inclusion using
\texttt{TransGen.mono}, and then applying the acyclicity hypothesis. This lemma is the
workhorse of the SC~$\Rightarrow$~TSO comparison in Section~\ref{sec:comparison}: once we
show that every edge in the TSO acyclicity constraint is also an edge in the SC constraint,
the result follows immediately.

We also prove a generalized variant, \texttt{acyclicMono\_trans}, which allows each edge
of~$r_1$ to correspond to a \emph{path} (rather than a single edge) in~$r_2$. This stronger
form is useful when the relations in the two models are not in direct subset relation but one
model's edges can be decomposed into multi-step paths in the other.

\paragraph{Acyclicity via Ranking Functions.}
For concrete litmus test executions, we sometimes need to show acyclicity of a specific
finite relation. A convenient technique is to exhibit a \emph{ranking function}
$f : \text{Event} \to \mathbb{N}$ that strictly increases along every edge:

\begin{lstlisting}
lemma acyclic_of_rank
  {r : SetRel Event Event}
  (f : Event → ℕ)
  (hf : ∀ a b, (a, b) ∈ r → f a < f b)
  : SetRel.Acyclic r
\end{lstlisting}

The proof proceeds by induction on the transitive closure: any path in~$r^+$ strictly
increases~$f$ at each step, so a cycle would require $f(a) < f(a)$, contradicting
\texttt{Nat.lt\_irrefl}. This lemma is particularly useful for the litmus test case studies
in Section~\ref{sec:snippets}, where the ranking function can often be defined directly from the event identifiers.

\paragraph{Other Supporting Lemmas.}
We proved several additional theorems that are used
throughout the case studies. The lemmas \texttt{strictPartialOrderImpliesAcyclic} and
{\texttt{AcyclicImpliesIrreflexive} establish the equivalence between the Cat-style
acyclicity judgment and the standard strict-order properties available in \texttt{Mathlib}.
The composition theorems \texttt{rf\_fr\_subset\_co} and \texttt{fr\_co\_subset\_fr} capture
the algebraic identities of the primitive relations: composing $\mathit{rf}$ then $\mathit{fr}$
yields $\mathit{co}$, and composing $\mathit{fr}$ then $\mathit{co}$ yields $\mathit{fr}$.
As a direct corollary, \texttt{co\_acyclic} follows from the well-formedness condition on
$\mathit{co}$ and \texttt{strictPartialOrderImpliesAcyclic}. Finally,
\texttt{internalImpliesPoOrPoMinusOne} states that two distinct same-thread events must be
related by program order in one direction, since $\mathit{po}$ is a total order per thread.
 
Together, these theorems form the proof infrastructure of LeanCat. In the following sections,
we demonstrate how they are applied in practice.
 

\section{Case Studies}
\label{subsection: Subsection of Results}
\subsection{Litmus Test Executions}
\label{sec:snippets}

We proved that SC is stronger than TSO and TSO is stronger than MIPS. We can use litmus tests to demonstrate this. Litmus tests are small concurrent programs with specific initial conditions and a query about whether a particular outcome is observable. They serve as a convenient way to distinguish memory models: if one model allows an outcome that another forbids, the models are provably different in strength.

\subsubsection{Store Buffering}

First, we use our classical Store Buffering (SB) example. The execution graph for this test is shown in Figure~\ref{fig:sb-fre-cycle}.

As shown in Figure~\ref{fig:sb-fre-cycle}, the execution contains a cycle that passes through two W$\to$R program-order edges. This behavior of the SB test is disallowed by Sequential Consistency, but allowed by x86-TSO, because the W$\to$R program-order pairs do not participate in the preserved program order under TSO, so the cycle in Figure~\ref{fig:sb-fre-cycle} is broken. MIPS likewise allows this outcome, because MIPS only preserves R$\to$W program-order pairs in the execution graph, so the W$\to$R pairs in the SB test again do not form a cycle. We proved this in \texttt{SB.lean}. In this example, we showed a case where SC forbids the outcome but both TSO and MIPS allow it.

\subsubsection{Message Passing}

\label{subsec:message_passing}

To further distinguish the ordering guarantees of x86-TSO and MIPS, we formalize the \textbf{Message Passing (MP)} litmus test in \texttt{MP.lean}. This test serves as a benchmark for the preservation of Store-to-Store ($W \to W$) and Load-to-Load ($R \to R$) ordering. The program structure is defined in Figure~\ref{fig:mp-program}, and the candidate execution graph representing the outcome $a = 1$ and $b = 0$ is depicted in Figure~\ref{fig:mp-rfe-cycle}.

\label{fig:mp-rfe-cycle}
\end{figure}

Under Sequential Consistency (SC), this outcome is disallowed as it violates the foundational requirement of global instruction interleaving. Within the x86-TSO model, the outcome is also forbidden due to the axiomatic invariant that both Store-Store and Load-Load program-order pairs must be preserved. Consequently, as illustrated in Figure~\ref{fig:mp-rfe-cycle}, the $po$ edges in both threads are included in the preserved program order, and the resulting cycle is detected as a violation of the acyclicity condition.

In contrast, the MIPS model permits this behavior. As a relaxed architecture, MIPS does not enforce ordering between independent memory accesses unless an explicit synchronization barrier is present. In our \texttt{LeanCat} formalization, these $po$ edges are excluded from the \textit{ppo} relation for MIPS, effectively breaking the cycle shown in Figure~\ref{fig:mp-rfe-cycle}. The acyclicity check therefore passes, providing a machine-checked proof that MIPS allows weak behaviors that TSO strictly forbids.

  is disallowed under MIPS and forbidden under x86-TSO.}
\label{fig:lb-rfe-cycle}
\end{figure}

Under Sequential Consistency (SC), this outcome is impossible: every valid interleaving requires at least one read to occur before the corresponding write, ensuring that at least one thread observes the initial value 0. Within the x86-TSO model, the outcome is similarly forbidden. As Figure~\ref{fig:lb-rfe-cycle} demonstrates, the cycle relies on $R \to W$ program-order edges; because TSO maintains a foundational verification of $R \to W$ ordering, this cycle cannot be eliminated. Furthermore, under MIPS, the outcome is also disallowed. Although MIPS is a relaxed architecture, it preserves $R \to W$ program-order pairs to prevent "out-of-thin-air" behaviors. Consequently, the \textit{po} edges in the cycle of Figure~\ref{fig:lb-rfe-cycle} contribute to the preserved program order, and the acyclicity check correctly identifies the execution as disallowed. We have verified this property through our formalization in LB.lean.

Together, these two litmus tests confirm the strict hierarchy: SC $\subset$ TSO $\subset$ MIPS. The SB test separates SC from TSO (and MIPS), while the MP test separates TSO from MIPS, and all of them forbid the \href{https://github.com/nek0las/lean-cats/blob/main/LeanCats/Linux/LB.lean}{LB.lean}.

\subsubsection{LKMM Litmus Tests}

We also formalize litmus tests for the Linux Kernel Memory Model (LKMM). The LKMM is a significantly more complex memory model than the generic axiomatic models discussed above, as it is designed to capture the specific semantics of the Linux kernel's concurrency primitives. It distinguishes between different access types --- such as \texttt{ONCE}, \texttt{RELEASE}, \texttt{ACQUIRE}, and \texttt{MB} --- and includes fence instructions like \texttt{wmb}, \texttt{rmb}, and \texttt{barrier}. The model defines several acyclicity conditions: \emph{coherence}, which requires the union of same-location program order and communication edges to be acyclic; \emph{atomic}, which ensures read-modify-write operations are not interleaved; \emph{happens-before}, built from preserved program order, reads-from-external edges, and internal propagation; and \emph{propagation}, which governs the visibility of writes across threads through cumulative fences.

We present a Coherence of Read-Read (CoRR) litmus test under the LKMM, formalized in \texttt{CoRR.lean}. The program is shown in Figure~\ref{fig:lkmm-corr-program}, and the candidate execution graph is shown in Figure~\ref{fig:lkmm-corr-cycle}.

In this test, all accesses are annotated with \texttt{ONCE}, the weakest marked access type in the LKMM. P0 performs a single write \texttt{WRITE\_ONCE(*x, 1)}, while P1 reads $X$ twice using \texttt{READ\_ONCE}. The question is whether P1 can first observe the updated value $r_0 = 1$ and then observe the initial value $r_1 = 0$.

As shown in Figure~\ref{fig:lkmm-corr-cycle}, if this outcome were allowed, the execution would contain a cycle in $po\_loc \cup com$. The two reads on P1 access the same location $X$ and are related by $po\_loc$. The second read $R(X) = 0$ reads from the initial write, which is coherence-before $W(X) = 1$, giving rise to a from-read ($fr$) edge from $R(X) = 0$ to $W(X) = 1$. Finally, $W(X) = 1$ feeds back to the first read $R(X) = 1$ via a reads-from ($rf$) edge.

This cycle violates the LKMM's coherence condition, which requires $\mathit{po\_loc} \cup \mathit{com}$ to be acyclic, where $\mathit{com} = \mathit{rf} \cup \mathit{co} \cup \mathit{fr}$. Since coherence is the most basic consistency requirement in the LKMM and does not depend on the access annotations or fence instructions, this outcome is forbidden regardless of the access types used. We proved this in \texttt{CoRR\_poonceonce\_Once.lean} by explicitly constructing the cycle in the coherence relation and deriving a contradiction.

\subsection{Comparing Memory Models}
\label{sec:comparison}

To check whether a memory model is weaker or stronger than another, we can use LeanCat 
to verify it mechanically. As we discussed in Section~\ref{sec:math-background}, a model 
$M_1$ is stronger than $M_2$ if every execution allowed by $M_1$ is also allowed by $M_2$, 
that is, $M_2$ permits at least as many behaviors as $M_1$. Formally, we require
%
\begin{equation}
  \{X \mid X \in S_c \text{ and } \mathrm{allowed}_{\mathrm{strict}}(X)\}
  \;\subseteq\;
  \{X \mid X \in S_c \text{ and } \mathrm{allowed}_{\mathrm{weak}}(X)\}.
\end{equation}

\subsubsection{Comparing SC, MIPS, and X86-TSO}

We compare three memory models: Sequential Consistency, Total Store Order (x86-TSO), 
and MIPS. Although these models are relatively simple, x86-TSO and MIPS are 
widely used in practice.

\paragraph{X86-TSO Semantics.}
We defined TSO in Figure~\ref{fig:cat-tso}, but that definition is incomplete. Two 
additional properties are required: \emph{atomicity} and \emph{sequential consistency 
per location}. The complete x86-TSO definition is therefore:

\begin{lstlisting}[]
let com = rf | fr | co
acyclic po_loc | com as uniproc

empty rmw & (fre;coe) as atomic

let xppo = ((W*W) | (R*W) | (R*R)) & po
let At = domain(rmw) | range(rmw)
let implied = po;[At | F] | [At | F];po
acyclic (implied | xppo | rfe | fr | co) as tso
\end{lstlisting}

The \texttt{uniproc} constraint enforces sequential consistency per location, which is a 
fundamental requirement for concurrent programming: all operations that access the same 
memory location must appear to execute in a sequentially consistent order. Without this 
constraint, concurrent accesses to the same variable could produce inconsistent results, 
corrupting the observed state of any program that relies on them. The \texttt{atomic} 
constraint ensures that read-modify-write operations are indivisible: no other write may 
be interleaved between the read and write parts of an atomic operation. Together, these 
two properties complete the x86-TSO model. Formally, they are expressed as:
%
\begin{align}
  &\texttt{empty}\ \mathit{rmw}\ \&\ (\mathit{fre};\mathit{coe})\ \texttt{as atomic}, \\
  &\texttt{acyclic}\ \mathit{po\_loc}\ |\ \mathit{com}\ \texttt{as uniproc}.
\end{align}

\paragraph{MIPS Semantics.}
The MIPS memory model is more relaxed than x86-TSO. Specifically, MIPS only preserves 
program order for pairs of operations in which the first event is a read; all other 
program-order pairs may be reordered. As a consequence, MIPS permits the execution in 
Example~\ref{fig:mp-program}, since it does not include the $W{\times}W \cap \mathit{po}$ 
constraint. We define the MIPS semantics as follows:

\begin{lstlisting}[]
let fencerel(r) = (po & (_ * r)) ; po
let com = rf | fr | co
acyclic po_loc | com as uniproc

let sync = try fencerel(SYNC) with 0

empty rmw & (fre;coe) as atomic

let ppo = po & (R * M) | sync

let ghb = ppo | rfe | fr | co
acyclic ghb as pso
\end{lstlisting}

In LeanCat, we use the \texttt{defcat} tactic to load Cat files into Lean~4. This produces 
a predicate for each model that takes a candidate execution as input, reflecting the 
general proof goal: for all candidate executions, one architecture is weaker than another.

\begin{lstlisting}[]
defcat <"sc.cat">
defcat <"tso.cat">
defcat <"mips.cat">
\end{lstlisting}

\noindent
This parses the Cat files and elaborates them into the Lean namespaces \texttt{SC}, 
\texttt{TSO}, and \texttt{MIPS}, giving us predicates derived from their respective 
definitions. Each predicate holds when the candidate execution \texttt{X} is 
\emph{allowed} under the corresponding model.

The theorems we wish to prove are: for every candidate execution $X$, if $X$ is allowed 
under SC then $X$ is also allowed under TSO, and if $X$ is allowed under TSO then $X$ 
is also allowed under MIPS.

\begin{lstlisting}[]
theorem scvtso
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : sc.sc evts X -> tso.tso evts X
\end{lstlisting}

\begin{lstlisting}[]
theorem mipsWeakerThanTso
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : tsox.tso evts X -> mips.pso evts X
\end{lstlisting}

When comparing MIPS and x86-TSO, we omit the \texttt{atomic} and \texttt{uniproc} 
constraints, as they are identical in both models and can be discharged directly.

Both proofs follow the same structure: we apply \texttt{acyclicMono} or 
\texttt{acyclicMono\_trans} to reduce the goal to a subset check, and then perform 
a case analysis to route each edge to the appropriate component of the larger relation.

For \texttt{scvtso}, we show that every edge in the TSO acyclicity constraint 
$(\mathit{implied} \cup \mathit{xppo} \cup \mathit{rfe} \cup \mathit{fr} \cup 
\mathit{co})$ is also an edge in the SC constraint $(\mathit{po} \cup \mathit{com})$. 
This holds because \texttt{implied} and \texttt{xppo} edges are both sub-relations 
of $\mathit{po}$, while $\mathit{rfe}$, $\mathit{fr}$, and $\mathit{co}$ are all 
sub-relations of $\mathit{com}$. The proof can be found at \href{https://github.com/nek0las/lean-cats/blob/main/LeanCats/TsoWeakerThanSc.lean}{LeanCats/TsoWeakerThanSc.lean}.

For \texttt{mipsWeakerThanTso}, we show that every edge in the MIPS constraint 
$(\mathit{ppo}_{\mathrm{MIPS}} \cup \mathit{rfe} \cup \mathit{fr} \cup \mathit{co})$ 
can be realized as a path in the TSO constraint. The $\mathit{rfe}$, $\mathit{fr}$, 
and $\mathit{co}$ components are shared between the two models. For 
$\mathit{ppo}_{\mathrm{MIPS}}$, a read--memory-access $\mathit{po}$ edge maps 
directly to \texttt{xppo}, and a \texttt{sync} edge maps to \texttt{implied} because 
every MIPS SYNC event is also a fence event in TSO. The proof can be found at \href{https://github.com/nek0las/lean-cats/blob/main/LeanCats/mipsWeakerThanTSO.lean}{LeanCats/mipsWeakerThanTSO.lean}.

\subsubsection{Comparing BPF and LKMM}

As discussed in Section~\ref{Background:BPF}, the BPF memory model shares the same high-level structure as the LKMM: both enforce coherence, happens-before, propagation, and atomicity constraints, and both capture their ordering requirements through \textit{ppo} and \textit{prop} relations. However, their synchronization primitives are expressed at different levels of abstraction. The LKMM is defined in terms of Linux kernel C primitives --- \texttt{smp\_mb}, \texttt{smp\_wmb}, \texttt{smp\_rmb}, acquire, and release --- whereas the BPF model is defined closer to the assembly level, using AArch64-style atomic memory operations (\texttt{po\_amo\_fetch}) and explicit acquire/release annotations (\texttt{AQ}/\texttt{RL}). Establishing a formal mapping between these two fence systems requires a careful translation between the two frameworks, and to our knowledge no such mapping has been proposed in the literature. Due to this gap and the time constraints of this project, we were unable to prove any properties involving fence orderings between BPF and LKMM.

The BPF model is defined in Figure~\ref{fig:bpf-definition}.

\begin{figure}[t]
  \centering
  \begin{minipage}{0.95\linewidth}
\begin{verbatim}
let po_amo_fetch = ([M];po;RMW) | (RMW;po;[M])
let load_acquire = ([AQ];po;[M])
let store_release = ([M];po;[RL])
let rcpc = load_acquire | store_release

let addr_dep = [R];addr;[M]
let data_dep = [R];data;[W]
let ctrl_dep = [R];ctrl;[W]

let com = co | rf | fr

let ppo =
    po_amo_fetch | rcpc
  | addr_dep | data_dep | ctrl_dep
  | [M];(addr|data);[W];rfi;[R]
  | [M];addr;[M];po;[W]
  | (coi | fri)

let A-cumul = rfe? ; (po_amo_fetch | store_release)
let prop = (coe | fre)? ; A-cumul* ; rfe?

acyclic com | po-loc       as Coherence
let hb = ppo | rfe | ((prop \ id) & int)
acyclic hb                 as Happens-before
let pb = prop ; po_amo_fetch ; hb*

\end{verbatim}

Despite the fence gap, we are able to prove a partial comparison: restricting attention to the fence-free fragment of both models, BPF is stronger than LKMM. Concretely, we show that every candidate execution allowed by the BPF happens-before constraint (excluding fence-induced orderings) is also allowed by the LKMM. This result is expected on safety grounds: since BPF programs execute inside the Linux kernel, they must not exhibit behaviors that the LKMM prohibits. Formally, the theorem we prove is

\begin{lstlisting}[]
theorem bpfStrongerThanLkmm
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : bpf.Happens_before evts X -> lkmm.hb evts X
\end{lstlisting}

The proof follows the same structure as the SC-TSO and TSO-MIPS comparisons: we apply \texttt{acyclicMono} and reduce the goal to showing that every edge in the LKMM's \textit{ppo} is also an edge in BPF's \textit{hb}. The \textit{ppo} of the LKMM decomposes into six cases, of which four admit a direct mapping.

\begin{itemize}
  \item \textbf{Address dependencies.} The LKMM's $\mathit{addr}$ edge maps to BPF's $\mathit{addr\_dep} = [R];\mathit{addr};[M]$. The well-formedness hypothesis \texttt{haddr\_wf} supplies the required type guards $a \in R$ and $b \in M$.

  \item \textbf{Dependency-rfi chains.} The LKMM's $(\mathit{addr} \cup \mathit{data});\mathit{rfi}$ maps to BPF's $[M];(\mathit{addr} \mid \mathit{data});[W];\mathit{rfi};[R]$. The intermediate write witness $w \in W$ and the endpoint $b \in R$ are obtained from the well-typedness of the reads-from relation (\texttt{rfInst.wellTyped}), and $a \in M$ follows from \texttt{haddr\_wf} or \texttt{hdata\_wf}.

  \item \textbf{Read-write dependencies.} The LKMM's $\mathit{rwdep} = (\mathit{dep} \cup \mathit{ctrl}) \cap (R \times W)$ is handled by a case split on the dependency type. Each case matches a corresponding BPF component --- $\mathit{addr\_dep}$, $\mathit{data\_dep}$, or $\mathit{ctrl\_dep}$ --- whose type guards are already satisfied by $\mathit{rwdep}$'s $(R \times W)$ constraint.

  \item \textbf{Coherence-internal overwrites.} The LKMM's $(\mathit{co} \cup \mathit{fr}) \cap \mathit{int}$ maps directly to BPF's $\mathit{coi} \cup \mathit{fri}$, since $\mathit{coi} = \mathit{co} \cap \mathit{int}$ and $\mathit{fri} = \mathit{fr} \cap \mathit{int}$ by definition.
\end{itemize}

The remaining two cases --- $\mathit{addr};[\mathit{Plain}];\mathit{wmb}$ and the full \textit{fence} ordering --- require a formal correspondence between the LKMM's fence primitives and BPF's \texttt{po\_amo\_fetch}. The core difficulty is that the \texttt{CandidateExecution} in LeanCat encodes fence relations (\texttt{mb}, \texttt{wmb}, \texttt{fence}) as opaque \texttt{SetRel} fields with no structural connection to BPF's \texttt{po\_amo\_fetch = ([M];po;RMW) | (RMW;po;[M])}. Without an explicit mapping axiom relating these two, neither case can be discharged. These cases are therefore left with \texttt{sorry} placeholders in the Lean development, and completing them is left as future work.

\section{Proof Engineering and Heuristic Automation}

With the emergence of high-capability models such as Opus and Codex, AI is capable of addressing the tedious overhead of formalization, though it can also introduce unvalidated heuristic errors. An interactive proof assistant is fundamentally distinct because it maintains strict foundational verification; it can provide precise feedback to the LLM when it detects errors, enabling an iterative cycle of refinement. Consequently, the precise formulation of theorem statements and the demarcation between AI-generated code and the trusted parts of the development become critical considerations.

In this framework, our contribution is the provision of a trusted infrastructure that translates the Cat language into Lean 4 directly, ensuring reusability across many memory models. Traditionally, formalizers must write corresponding proofs for each model manually, incurring significant overhead in redundant boilerplate and common theorems. While one could task an LLM with reasoning about a Cat file directly without such a framework, we must recognize that its heuristic nature can mislead humans by introducing incorrect assumptions for the models. This is why it is essential to maintain a sharp separation between the trusted core and the LLM-generated code.

In our project, we relegate these tedious proofs to automation. For any given litmus test, a cycle is often easily identified by inspection, but the formal mechanization is notably labor-intensive. It requires the manual construction of transitive closure (\texttt{TransGen}) chains and the composition of relations to demonstrate that they indeed form a cycle.

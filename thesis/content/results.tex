\section{Visualization}

In Lean 4, we use the \emph{InfoView} panel on the right-hand side of Visual Studio Code. We typically rely on this panel (via the Lean VSCode extension) to inspect the current goals and track the progress of a proof.

In our development, we use type classes together with \texttt{Set} and \texttt{RelSet} to define candidate executions. However, this representation can be difficult to reason about from a human perspective. Essentially, graphs in Lean 4 are encoded in a way that is convenient for machine processing, but not for human intuition. It would be much easier to understand the structure—especially to identify cycles—if we could directly visualize the graph.

To achieve this visualization, we use the tool \href{https://github.com/leanprover-community/ProofWidgets4}{ProofWidgets}. Broadly speaking, it provides APIs that allow us to visualize data and proofs interactively. Let us look at an example of how it works in LeanCat.

The following program represents the X86 \emph{Store Buffering (SB)} litmus test (from Figure~\ref{fig:litmus-test}):

\begin{verbatim}
/-! ### X86 SB (Store Buffering) Litmus Test -/

private def sbEvents : Array Event :=
  #[initWx, initWy, inst1writeX, inst2readY, inst3writeY, inst4readX]

private def sbRf : Array (Event × Event) :=
  #[(initWy, inst2readY), (initWx, inst4readX)]

private def sbCo : Array (Event × Event) :=
  #[(initWx, inst1writeX), (initWy, inst3writeY)]

/-- Concrete SB litmus-test execution, built from the events and relations
defined in `Litmus.lean`. -/
def sbExecution : ConcreteExecution := {
  events     := sbEvents
  locName    := fun n => n
  threadName := fun n => n
  po         := computeDirectPo sbEvents
  rf         := sbRf
  co         := sbCo
  fr         := computeFr sbRf sbCo
  rmw        := #[]
}
\end{verbatim}

After defining the base relations (\texttt{rf}, \texttt{co}, and derived \texttt{fr}), we construct a candidate execution. The \texttt{co} relation can be derived directly from the litmus test, and \texttt{fr} is computed from \texttt{rf} and \texttt{co}.

Once the candidate execution is defined, we can use the command:

\begin{verbatim}
#html toOrderedHtml sbExecution
\end{verbatim}

This produces a clear graphical representation of the execution. From this graph, we can easily observe the presence of a cycle (W(x) = 1 -> R(y) -> W(y) = 1 -> R(x)), which indicates that the execution is disallowed under the memory model. This visualization can then serve as a useful starting point for proving the result formally.

\section{Theorems}
\subsection{Theorems}

To support the case studies, we proved a collection of library theorems about binary relations,
acyclicity, and the primitive relations of candidate executions. These theorems bridge the gap
between Lean~4's existing \texttt{Mathlib} library and the specific needs of weak memory
model reasoning. We highlight the two most important results in detail and briefly summarise
the remaining supporting lemmas.

\paragraph{Acyclicity monotonicity.}
The central proof technique for comparing memory models is \emph{monotonicity}: if a
relation~$r$ is acyclic and~$s \subseteq r$, then~$s$ is also acyclic. We formalise this as
\texttt{acyclicMono}:

\begin{lstlisting}
lemma acyclicMono
  {r₁ r₂ : SetRel Event Event}
  (hacyc : SetRel.Acyclic r₂)
  (hsub : ∀ a b, (a, b) ∈ r₁ → (a, b) ∈ r₂)
  : SetRel.Acyclic r₁
\end{lstlisting}

The proof proceeds by lifting the edge-wise inclusion to a transitive-closure inclusion using
\texttt{TransGen.mono}, and then applying the acyclicity hypothesis. This lemma is the
workhorse of the SC~$\Rightarrow$~TSO comparison in Section~\ref{sec:comparison}: once we
show that every edge in the TSO acyclicity constraint is also an edge in the SC constraint,
the result follows immediately.

We also prove a generalised variant, \texttt{acyclicMono\_trans}, which allows each edge
of~$r_1$ to correspond to a \emph{path} (rather than a single edge) in~$r_2$. This stronger
form is useful when the relations in the two models are not in direct subset relation but one
model's edges can be decomposed into multi-step paths in the other.

\paragraph{Acyclicity via ranking functions.}
For concrete litmus test executions, we sometimes need to show acyclicity of a specific
finite relation. A convenient technique is to exhibit a \emph{ranking function}
$f : \text{Event} \to \mathbb{N}$ that strictly increases along every edge:

\begin{lstlisting}
lemma acyclic_of_rank
  {r : SetRel Event Event}
  (f : Event → ℕ)
  (hf : ∀ a b, (a, b) ∈ r → f a < f b)
  : SetRel.Acyclic r
\end{lstlisting}

The proof proceeds by induction on the transitive closure: any path in~$r^+$ strictly
increases~$f$ at each step, so a cycle would require $f(a) < f(a)$, contradicting
\texttt{Nat.lt\_irrefl}. This lemma is particularly useful for the litmus test case studies
in Section~\ref{sec:snippets}, where the ranking function can often be defined directly from
the event identifiers.

\paragraph{Other supporting lemmas.}
We proved several additional theorems that are used
throughout the case studies. The lemmas \texttt{strictPartialOrderImpliesAcyclic} and
{\texttt{AcyclicImpliesIrreflexive} establish the equivalence between the Cat-style
acyclicity judgment and the standard strict-order properties available in \texttt{Mathlib}.
The composition theorems \texttt{rf\_fr\_subset\_co} and \texttt{fr\_co\_subset\_fr} capture
the algebraic identities of the primitive relations: composing $\mathit{rf}$ then $\mathit{fr}$
yields $\mathit{co}$, and composing $\mathit{fr}$ then $\mathit{co}$ yields $\mathit{fr}$.
As a direct corollary, \texttt{co\_acyclic} follows from the wellformedness condition on
$\mathit{co}$ and \texttt{strictPartialOrderImpliesAcyclic}. Finally,
\texttt{internalImpliesPoOrPoMinusOne} states that two distinct same-thread events must be
related by program order in one direction, since $\mathit{po}$ is a total order per thread.
 
Together, these theorems form the proof infrastructure of LeanCat. In the following sections,
we demonstrate how they are applied in practice.

\subsection{Program Snippets Check}

We proved that SC is stronger than TSO and TSO is stronger than MIPS. We can use litmus tests to demonstrate this. Litmus tests are small concurrent programs with specific initial conditions and a query about whether a particular outcome is observable. They serve as a convenient way to distinguish memory models: if one model allows an outcome that another forbids, the models are provably different in strength.

First, we use our classical Store Buffering (SB) example. The execution graph for this test is shown in Figure~\ref{fig:sb-fre-cycle}.

As shown in Figure~\ref{fig:sb-fre-cycle}, the execution contains a cycle that passes through two W$\to$R program-order edges. This behaviour of the SB test is disallowed by Sequential Consistency, but allowed by x86-TSO, because the W$\to$R program-order pairs do not participate in the preserved program order under TSO, so the cycle in Figure~\ref{fig:sb-fre-cycle} is broken. MIPS likewise allows this outcome, because MIPS only preserves R$\to$W program-order pairs in the execution graph, so the W$\to$R pairs in the SB test again do not form a cycle. We proved this in \texttt{SB.lean}. In this example, we showed a case where SC forbids the outcome but both TSO and MIPS allow it.

Under SC, this outcome is impossible: every interleaving requires at least one read to occur before the corresponding write, so at least one thread must read the initial value~0. Under TSO, the outcome is also forbidden. As Figure~\ref{fig:lb-rfe-cycle} illustrates, the cycle involves R$\to$W program-order edges, and TSO preserves R$\to$W ordering, so the cycle cannot be eliminated. However, under MIPS, the outcome is allowed. MIPS does not preserve R$\to$W program-order pairs, so neither of the \textit{po} edges in the cycle of Figure~\ref{fig:lb-rfe-cycle} contributes to the preserved program order, and the acyclicity check passes. We proved this in \texttt{LB.lean}.

Together, these two litmus tests confirm the strict hierarchy: SC $\subset$ TSO $\subset$ MIPS. The SB test separates SC from TSO (and MIPS), while the LB test separates TSO from MIPS.

We also formalize litmus tests for the Linux Kernel Memory Model (LKMM). The LKMM is a significantly more complex memory model than the generic axiomatic models discussed above, as it is designed to capture the specific semantics of the Linux kernel's concurrency primitives. It distinguishes between different access types --- such as \texttt{ONCE}, \texttt{RELEASE}, \texttt{ACQUIRE}, and \texttt{MB} --- and includes fence instructions like \texttt{wmb}, \texttt{rmb}, and \texttt{barrier}. The model defines several acyclicity conditions: \emph{coherence}, which requires the union of same-location program order and communication edges to be acyclic; \emph{atomic}, which ensures read-modify-write operations are not interleaved; \emph{happens-before}, built from preserved program order, reads-from-external edges, and internal propagation; and \emph{propagation}, which governs the visibility of writes across threads through cumulative fences.

In this test, all accesses are annotated with \texttt{ONCE}, the weakest marked access type in the LKMM. P0 performs a single write \texttt{WRITE\_ONCE(*x, 1)}, while P1 reads $X$ twice using \texttt{READ\_ONCE}. The question is whether P1 can first observe the updated value $r_0 = 1$ and then observe the initial value $r_1 = 0$.

As shown in Figure~\ref{fig:lkmm-corr-cycle}, if this outcome were allowed, the execution would contain a cycle in $po\_loc \cup com$. The two reads on P1 access the same location $X$ and are related by $po\_loc$. The second read $R(X) = 0$ reads from the initial write, which is coherence-before $W(X) = 1$, giving rise to a from-read ($fr$) edge from $R(X) = 0$ to $W(X) = 1$. Finally, $W(X) = 1$ feeds back to the first read $R(X) = 1$ via a reads-from ($rf$) edge.

This cycle violates the LKMM's coherence condition, which requires $\mathit{po\_loc} \cup \mathit{com}$ to be acyclic, where $\mathit{com} = \mathit{rf} \cup \mathit{co} \cup \mathit{fr}$. Since coherence is the most basic consistency requirement in the LKMM and does not depend on the access annotations or fence instructions, this outcome is forbidden regardless of the access types used. We proved this in \texttt{CoRR\_poonceonce\_Once.lean} by explicitly constructing the cycle in the coherence relation and deriving a contradiction.

\subsection{Comparison between different models for the same architecture}

To check whether a memory model is weaker or stronger than another, we can use LeanCat 
to verify it mechanically. As we discussed in Section~\ref{sec:math-background}, a model 
$M_1$ is stronger than $M_2$ if every execution allowed by $M_1$ is also allowed by $M_2$, 
that is, $M_2$ permits at least as many behaviors as $M_1$. Formally, we require
%
\begin{equation}
  \{X \mid X \in S_c \text{ and } \mathrm{allowed}_{\mathrm{strict}}(X)\}
  \;\subseteq\;
  \{X \mid X \in S_c \text{ and } \mathrm{allowed}_{\mathrm{weak}}(X)\}.
\end{equation}

We already showed in Section~\ref{sec:sc-tso} that TSO is defined by relaxing SC: it removes 
the write--read constraint from the acyclicity condition, allowing processors to reorder a 
write followed by a read. Intuitively, this means TSO should allow \emph{strictly more} 
behaviors than SC --- every execution that SC accepts, TSO must also accept. In LeanCat, 
we can verify this claim formally.

We first load both models using the \texttt{defcat} command:

\begin{lstlisting}[]
defcat <"sc.cat">
defcat <"tso.cat">
\end{lstlisting}

\noindent
This parses the Cat files and elaborates them into the Lean namespaces \texttt{SC} and 
\texttt{TSO}, giving us the predicates \texttt{sc.sc evts X} and \texttt{tso.tso evts X}, 
each of which holds when the candidate execution \texttt{X} is \emph{allowed} under the 
respective model.

The theorem we want to prove is: for every candidate execution $X$, if $X$ is allowed 
under SC, then $X$ is also allowed under TSO.

\begin{lstlisting}[]
theorem scvtso
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : sc.sc evts X -> tso.tso evts X
\end{lstlisting}

Recall from Section~\ref{sec:sc-tso} that SC requires
%
\begin{equation}
  \mathtt{acyclic}\ (po \mid com),
\end{equation}
%
while TSO requires
%
\begin{equation}
  \mathtt{acyclic}\ (\mathit{implied} \mid \mathit{xppo} \mid \mathit{rfe} \mid fr \mid co),
\end{equation}
%
where $\mathit{xppo} = ((W \times W) \cup (R \times W) \cup (R \times R)) \cap po$ strips 
out write--read pairs from $po$, and $\mathit{implied}$ re-introduces ordering only around 
atomic operations and fences (see Figure~\ref{fig:tso}). The key insight is that every edge 
in the TSO acyclicity constraint is already present --- possibly in a weaker or equal form 
--- in the SC constraint. The proof therefore proceeds by \emph{monotonicity}: if 
$po \cup com$ is acyclic, then any sub-relation of it is also acyclic.

The proof unfolds in two stages. First, we unfold both model predicates and reduce the 
goal to showing that every edge in the TSO relation is already an edge in the SC relation:

\begin{lstlisting}[]
by
unfold sc.sc
simp
intro sc
apply acyclicMono sc
\end{lstlisting}

\noindent
Here \texttt{acyclicMono} is a library lemma in LeanCat stating that if a relation $r$ is 
acyclic and $s \subseteq r$, then $s$ is also acyclic. It captures exactly the monotonicity 
argument above. We discharge each case of the TSO relation by locating the corresponding edge in 
$po \cup com$. The case split follows the structure of the TSO constraint:


\textbf{\texttt{implied} edges} arise from $po$ composed with an atomic or fence 
  event. These reduce either to a direct $po$ edge or to a $po$-transitivity step, both of 
  which are contained in SC's $po$ component:

\begin{lstlisting}[]
· rcases hImplied with ⟨mid, hpo_amid, htail⟩
  ...
  exact Or.inl (X.prePo _ _ _ hpo_amid hpo_xb)
\end{lstlisting}

\textbf{\texttt{xppo} edges} are a subset of $po$ restricted to non-write--read 
  pairs, so they map directly into SC's $po$ component:

\begin{lstlisting}[]
· exact Or.inl hxppo.2
\end{lstlisting}

\textbf{\texttt{rfe}, \texttt{fr}, and \texttt{co} edges} are all part of 
  $com = rf \cup fr \cup co$ in SC, so they map into SC's $com$ component:

\begin{lstlisting}[]
· exact Or.inr (Or.inr (Or.inl hrfe.1))
· exact Or.inr (Or.inr (Or.inl hfr))
· exact Or.inr (Or.inr (Or.inr hco))
\end{lstlisting}

Every branch closes successfully, and Lean's kernel accepts the proof term. This gives us 
a machine-checked certificate that SC is strictly stronger than TSO: any execution the SC 
model permits, TSO must also permit, but not vice versa. The store-buffering execution 
from Figure~\ref{fig:sb}, where both reads observe $0$, is forbidden under SC yet allowed 
under TSO, which confirms that the inclusion is strict.

This workflow --- \texttt{defcat}, state the inclusion theorem, apply \texttt{acyclicMono}, 
case-split on the relation structure --- generalises directly to other model comparisons. 
For example, one could compare LKMM against TSO for the architectures it supports, or 
verify that adding a fence primitive to a custom model makes it strictly stronger than the 
baseline. LeanCat reduces each such comparison to the same pattern, so the mechanization 
effort scales with the structural difference between the two models rather than with the 
complexity of either model individually.

\subsubsection{Counterexample: TSO allows an execution that SC forbids}

To make the comparison complete, we also need to show the other direction: that TSO is 
\emph{strictly} weaker than SC, not merely weaker. That is, there exists a candidate 
execution that TSO permits but SC forbids. We already introduced this execution in 
Figure~\ref{fig:sb} --- it is the Store-Buffering execution where both reads observe $0$.

Let us return to the Store-Buffering litmus test from Figure~\ref{fig:sb}. Recall that 
both $X$ and $Y$ are initialized to $0$. Thread 0 writes $1$ to $X$ and then reads $Y$, 
while Thread 1 writes $1$ to $Y$ and then reads $X$. The specific outcome we are 
interested in is where both reads observe $0$, that is, $a = 0$ and $b = 0$.

\paragraph{Why SC forbids this execution.}
Under Sequential Consistency, we require
%
\begin{equation}
  \mathtt{acyclic}\ (po \cup com).
\end{equation}
%
In this execution, the fr relation arises because $R(Y) = 0$ reads the initial value of 
$Y$, which is later overwritten by $W(Y) = 1$, and similarly $R(X) = 0$ reads the 
initial value of $X$, which is later overwritten by $W(X) = 1$. This produces the 
following cycle:
%
\begin{equation}
  W(X) \xrightarrow{po} R(Y) \xrightarrow{fr} W(Y) \xrightarrow{po} R(X) 
  \xrightarrow{fr} W(X).
\end{equation}
%
The edges $W(X) \xrightarrow{po} R(Y)$ and $W(Y) \xrightarrow{po} R(X)$ are 
program-order edges, and the edges $R(Y) \xrightarrow{fr} W(Y)$ and 
$R(X) \xrightarrow{fr} W(X)$ are from-read edges. Together they form a cycle in 
$po \cup com$, which violates the acyclicity condition. SC therefore forbids this 
execution.

\paragraph{Why TSO allows this execution.}
Recall from Figure~\ref{fig:tso} that TSO requires
%
\begin{equation}
  \mathtt{acyclic}\ (\mathit{implied} \cup \mathit{xppo} \cup \mathit{rfe} 
  \cup fr \cup co),
\end{equation}
%
where $\mathit{xppo} = ((W \times W) \cup (R \times W) \cup (R \times R)) \cap po$ 
strips out write--read pairs from $po$. The critical edges $W(X) \xrightarrow{po} R(Y)$ 
and $W(Y) \xrightarrow{po} R(X)$ are both write--read pairs. They are therefore excluded 
from $\mathit{xppo}$ and do not appear in the TSO acyclicity constraint. Without these 
edges, the cycle breaks and the execution is allowed under TSO.

\paragraph{Mechanizing the counterexample in LeanCat.}
We construct this execution concretely in Lean 4, following the same pattern as the 
\texttt{sbExecution} definition from Section~\ref{sec:visualization}:

\begin{lstlisting}[]
private def sbEvents : Array Event :=
  #[initWx, initWy, inst1writeX, inst2readY, inst3writeY, inst4readX]

private def sbRf : Array (Event × Event) :=
  #[(initWx, inst2readY), (initWy, inst4readX)]

private def sbCo : Array (Event × Event) :=
  #[(initWx, inst1writeX), (initWy, inst3writeY)]

def sbExecution : ConcreteExecution := {
  events    := sbEvents
  po        := computeDirectPo sbEvents
  rf        := sbRf
  co        := sbCo
  fr        := computeFr sbRf sbCo
  rmw       := #[]
}
\end{lstlisting}

\noindent
Here the rf relation maps the initial writes \texttt{initWx} and \texttt{initWy} to 
the corresponding read events, meaning both reads observe the initial value $0$. The 
co relation records that the initial writes are overwritten by the thread writes, 
giving rise to the fr edges described above.

Once the candidate execution is defined, we can state and prove the two directions 
formally. The first theorem states that this execution is allowed under TSO:

\begin{lstlisting}[language=lean]
theorem sb_allowed_tso
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : tso.tso evts sbExecution := by
  unfold tso.tso
  simp [Acyclic]
  -- No cycle exists in (implied | xppo | rfe | fr | co)
  -- because the write-read po edges are excluded from xppo
  decide
\end{lstlisting}

\noindent
The second theorem states that the same execution is forbidden under SC:

\begin{lstlisting}[language=lean]
theorem sb_forbidden_sc
  (evts : Data.Events)
  (X : CandidateExecution evts)
  : ¬ sc.sc evts sbExecution := by
  unfold sc.sc
  simp [Acyclic]
  -- The cycle W(X) -po-> R(Y) -fr-> W(Y) -po-> R(X) -fr-> W(X)
  -- witnesses the violation of acyclicity
  decide
\end{lstlisting}

\noindent
Together, \texttt{scvtso}, \texttt{sb\_allowed\_tso}, and \texttt{sb\_forbidden\_sc} 
give us a complete machine-checked account of the relationship between SC and TSO. 
Every execution allowed by SC is also allowed by TSO, but not vice versa: the 
store-buffering execution with both reads observing $0$ is permitted by TSO yet 
forbidden by SC. This confirms that the inclusion is strict, and that TSO is 
\emph{strictly} weaker than SC.

\section{Proof Engineering in AI}
As the new programming LLM models, such as Opus and Codex, the AI is capable of solving tedious proofs, but it can also introduce hallucinations. The proof assistant is different because it has strict verification; it can give feedback to the LLM if it finds something wrong, and then the LLM can start iterations. So how to define a correct theorem statement becomes important, and how to separate the boundary between AI-generated code and the correct part is also important.

So in this framework, our contribution is to make a trusted framework that can convert the defined cat language into Lean 4 directly, and it is reusable for many memory models. As before, people usually write the corresponding proofs for each model, and there is a lot of boilerplate that can be reused, like some common theorems. With LLMs, we can also let the LLM prove the Cat file directly instead of using our framework, but we need to recognize that even though the LLM is powerful, it still has hallucinations; it can easily mislead humans by defining incorrect assumptions for the models. That is why we need to separate the trusted part and the LLM-generated code part.

In our project, we leave the tedious proofs to automation. For each litmus test, we can find the cycle easily just by inspection, but it is hard and tricky to write the proof, because we need to create the TransGen chain and then chain them together to show that they form a cycle.

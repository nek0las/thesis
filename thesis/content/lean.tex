\chapter{Background}
\label{chapter:Background}

In this chapter, we introduce weak memory, the Lean 4 programming language and its metaprogramming framework, and the mathematical background needed in later chapters. First, we introduce weak memory models through some basic examples and weak behaviors (Section \ref{Background:LKMM}).

\section{Mathematical Background}
\label{sec:math-background}

To understand the axiomatic memory models, we need to use \emph{binary relations}. Relationships between elements of two sets are represented using a structure called a binary relation. A binary relation from a set $A$ to a set $B$ is formally defined as a subset of the Cartesian product $A \times B$ \citep[p.~599]{rosen2012discrete}. There are two common methods for representing relations: using square matrices and using directed graphs \citep[p.~621]{rosen2012discrete}. In this thesis, we mainly use the second method, i.e., directed graphs, to represent binary relations.

We use the notation \(a \, R \, b\) to denote that \((a,b) \in R\), and \(a \not R b\) to denote that \((a,b) \notin R\) \citep[p.~600]{rosen2012discrete}.

In this thesis, the elements \(a\) and \(b\) are both \emph{events}, and these binary relations compose the directed execution graph, as described in Section~\ref{Background:AWMM}. Two relations can be composed together to obtain a new directed graph. Relation composition plays an important role in the \texttt{cat} language.

\paragraph{Relation Composition.}
Let $R$ be a relation from a set $A$ to a set $B$, and let $S$ be a relation from $B$ to a set $C$. The composite of $R$ and $S$ is the relation consisting of ordered pairs $(a, c)$, where $a \in A, c \in C$, and for which there exists an element $b \in B$ such that $(a, b) \in R$ and $(b, c) \in S$ \citep[p.~607]{rosen2012discrete}. We denote the composite of $R$ and $S$ by $S ; R$ \citep[p.~607]{rosen2012discrete}.

Computing the composite of two relations requires finding elements that are the second component of ordered pairs in the first relation and the first component of ordered pairs in the second relation. This property is widely used in many memory models. For example, there is a relation called \emph{fr} (from-read), which represents that a read instruction observes a value that is later overwritten by another write instruction; the write instructions to the same address are totally ordered by the coherence order.

In \texttt{cat}, we define
\[
fr = co ; rf^{-1},
\]
where \(rf^{-1}\) denotes the inverse of the relation \(rf\), meaning that the order of each pair is reversed. Intuitively, this composition connects a read event to a write event that comes later in coherence order than the write from which the read obtained its value.

After composing different relations, we can obtain a complete execution graph. As stated in Section~\ref{Background:AWMM}, we need to filter out disallowed executions using a set of judgments. The \texttt{cat} language provides three primitive judgments: \texttt{Acyclic}, \texttt{Irreflexive}, and \texttt{Empty}. Among them, \texttt{Acyclic} is the most important because our memory models are described by binary relations that compose the execution graph. A cycle (or circuit) in a directed graph is a path of length greater than zero that begins and ends at the same vertex \cite[pp.~665,~715]{rosen2012discrete}. A cycle in this graph usually means that some data that should have been read first was overwritten by later operations.

Models defined using binary relations can be viewed as directed graphs. When we illustrate them in a diagram, cycles are easy to identify by visually connecting the points. However, to represent this structure using binary relations, we need a formal way to express both direct connections and paths that return to earlier points. For this purpose, we rely on two properties: \emph{reflexivity} and \emph{transitive closure}.

\paragraph{Reflexivity.}
A relation $R$ on a set $A$ is called reflexive if $(a, a) \in R$ for every element $a \in A$ \citep[p.~602]{rosen2012discrete}. Formally:
\[
\forall a \in A, (a, a) \in R.
\]

\paragraph{Transitivity.}
A relation $R$ on a set $A$ is called transitive if whenever $(a, b) \in R$ and $(b, c) \in R$, then $(a, c) \in R$, for all $a, b, c \in A$ \citep[p.~604]{rosen2012discrete}. Formally:
\[
\forall a, b, c \in A, ((a, b) \in R \land (b, c) \in R) \Rightarrow (a, c) \in R.
\]

\paragraph{Transitive Closure.}
The transitive closure of a relation $R$ on a set $A$ is the smallest transitive relation that contains $R$ \citep[p.~628]{rosen2012discrete}. Formally, the transitive closure, often denoted by $R^+$, can be expressed as the union of all powers $R^n$ for $n \ge 1$ \citep[p.~631]{rosen2012discrete}:
\[
R^+ = \bigcup_{n=1}^{\infty} R^n.
\]
An ordered pair $(a, b)$ is in $R^+$ if and only if there is a path of length at least one from $a$ to $b$ in the directed graph representing $R$ \citep[p.~631]{rosen2012discrete}.

\paragraph{Reflexive-Transitive Closure.}
The reflexive-transitive closure, denoted by $R^*$, is the smallest relation that is both reflexive and transitive and contains $R$. It is also known as the connectivity relation and consists of all ordered pairs $(a, b)$ such that there is a path of any length (including length zero) from $a$ to $b$ \citep[p.~631]{rosen2012discrete}. It is formed by taking the union of the transitive closure $R^+$ and the diagonal relation $\Delta = \{(a, a) \mid a \in A\}$, which represents paths of length zero \citep[p.~631]{rosen2012discrete}:
\[
R^* = R^+ \cup \Delta = \bigcup_{n=0}^{\infty} R^n.
\]
In memory models, these closures are fundamental for defining global orders. For example, $\mathit{hb}^*$ (the reflexive-transitive closure of the happens-before relation defined in LKMM \ref{fig:lkmm-definitions}) connects two events if one happens before the other or if they are the same event.

\paragraph{Domain and Codomain.}
Let $R$ be a relation from a set $A$ to a set $B$.
The \emph{domain} of $R$, denoted by $\mathrm{dom}(R)$, is the set of all first components of ordered pairs in $R$~\citep[p.~607]{rosen2012discrete}:
\[
  \mathrm{dom}(R) = \{ a \in A \mid \exists\, b \in B.\; (a, b) \in R \}.
\]
The \emph{codomain} (or \emph{range}) of $R$, denoted by $\mathrm{codom}(R)$, is the set of all second components of ordered pairs in $R$~\citep[p.~607]{rosen2012discrete}:
\[
  \mathrm{codom}(R) = \{ b \in B \mid \exists\, a \in A.\; (a, b) \in R \}.
\]
In the context of memory models, these are used to identify events that participate in a relation.
For example, in the TSO model we define the set of atomic events as $\mathit{At} = \mathrm{dom}(\mathit{rmw}) \cup \mathrm{codom}(\mathit{rmw})$, which collects all events involved in a read-modify-write operation.

\paragraph{Paths in Directed Graphs.}
To represent these structures using binary relations, we rely on the concept of paths. A path from $a$ to $b$ in a directed graph $G$ is a sequence of edges $(x_0, x_1), (x_1, x_2), \dots, (x_{n-1}, x_n)$ in $G$, where $n$ is a nonnegative integer, $x_0 = a$, and $x_n = b$ \citep[p.~629]{rosen2012discrete}. The length of this path is $n$ \citep[p.~629]{rosen2012discrete}. The connectivity relation $R^*$ consists of the ordered pairs $(a, b)$ such that there is a path of length at least one from $a$ to $b$ in $R$ \citep[p.~631]{rosen2012discrete}. A fundamental result in relation theory is that the transitive closure of a relation $R$ is equal to the connectivity relation $R^*$ \citep[p.~631]{rosen2012discrete}.

\paragraph{Relation Inverse.}
Let $R$ be a relation from a set $A$ to a set $B$. The inverse relation, denoted by $R^{-1}$, is the relation from $B$ to $A$ consisting of the ordered pairs $(b, a)$ such that $(a, b) \in R$ \citep[p.~665]{rosen2012discrete}. Formally:
\[
R^{-1} = \{(b, a) \mid (a, b) \in R\}.
\]
This means that $b \, R^{-1} \, a$ if and only if $a \, R \, b$ \citep[p.~665]{rosen2012discrete}. In the context of memory models, the inverse of the reads-from relation, $\mathit{rf}^{-1}$, connects a read event back to the write event that provided its value.

\paragraph{Optional Relation (Reflexive Closure).}
The reflexive closure of a relation $R$ on a set $A$ is the smallest reflexive relation that contains $R$ \citep[p.~628]{rosen2012discrete}. In memory model formalisms such as the \texttt{cat} language, this is frequently referred to as an "optional" relation and is denoted by $R^?$ or $R?$. It is formed by taking the union of the relation $R$ and the diagonal relation $\Delta = \{(a, a) \mid a \in A\}$ \citep[p.~628]{rosen2012discrete}:
\[
R^? = R \cup \Delta.
\]
Intuitively, $R^?$ represents that two events are either related by $R$ or are the exact same event. This is used extensively in LKMM definitions (e.g., $\mathit{rfe}^?$) to indicate that a property may hold through a relation or via an identity transition.

\paragraph{Internal and External Relations.}
In the context of multi-threaded execution, every event $e$ is associated with a specific thread identifier. Let $E$ be the set of all events and $T$ be the set of thread identifiers. We define a function $\text{tid}: E \to T$ that maps each event to the thread that executed it.

The \emph{internal} relation, denoted by \texttt{int}, consists of pairs of events that belong to the same thread. Formally, it is the equivalence relation on $E$ induced by the thread identifier function \citep[p.~642]{rosen2012discrete}:
\[
\texttt{int} = \{ (e_1, e_2) \in E \times E \mid \text{tid}(e_1) = \text{tid}(e_2) \}.
\]

The \emph{external} relation, denoted by \texttt{ext}, consists of pairs of events that belong to different threads. It is defined as the complement of the internal relation with respect to the Cartesian product $E \times E$ \citep[p.~602]{rosen2012discrete}:
\[
\texttt{ext} = \{ (e_1, e_2) \in E \times E \mid \text{tid}(e_1) \neq \text{tid}(e_2) \}.
\]
Equivalently, $\texttt{ext} = (E \times E) \setminus \texttt{int}$. In memory models, these are used to derive specific sub-relations; for example, \texttt{rfi} (reads-from internal) is defined as $\mathit{rf} \cap \texttt{int}$, and \texttt{rfe} (reads-from external) is defined as $\mathit{rf} \cap \texttt{ext}$.

Relation composition, defined in Section~\ref{sec:math-background}, is central to formalizing memory models in Cat. As noted there, the \emph{fr} (from-read) relation is a key example: $\mathit{fr} = \mathit{co} \mathbin{;} \mathit{rf}^{-1}$.

\subsection{Comparing Memory Models}

From a high-level perspective, when comparing different models, we check whether one model allows more program behaviors than another. If it does, we say that this model is \emph{weaker} than the other. A program generates a set of candidate executions, denoted by \( S^c \), and the memory model filters these to determine which executions are allowed. A weaker memory model should include all executions allowed by a stricter memory model. Formally, we require:

\[
\{ X \mid X \in S^c \text{ and } \text{allowed}_{\text{strict}}(X) \}
\;\subseteq\;
\{ X \mid X \in S^c \text{ and } \text{allowed}_{\text{weak}}(X) \}.
\]

This captures the idea that every execution permitted by the stricter model must also be permitted by the weaker one.

\section{Weak Memory Model}
There was a widely held belief in the software community that we did not need to optimize software; we could focus on abstraction, and hardware improvements would deliver performance gains for us. This belief largely held during the single-threaded era, when increasing clock speeds and improved microarchitectures steadily accelerated programs. However, when single-core processors hit power and energy-efficiency limits, hardware designers shifted toward adding more cores to improve performance. At that point, synchronization and memory-consistency abstractions were not yet well defined for compiler and system engineers. Vendors relied heavily on experience and natural-language descriptions to explain hardware behavior. Because natural language is inherently ambiguous, it became difficult to reason precisely about many concurrent program behaviors.


The compiler will normally optimize the code aggressively for performance as long as the programmer can’t observe the difference between the unoptimized and optimized code, but in fact, the code is not equivalent. In a uniprocessor system it works fine, but when it comes to a multithreaded program the code can cause bugs. Also, the hardware itself will run the program out of order to optimize execution, or just speculatively execute. Optimizing compilers have great freedom in the way they translate source code to binary. They are allowed to apply transformations that add memory accesses, eliminate accesses, combine them, split them into pieces, or move them around~\cite{10.1145/2676726.2676995}.

Let’s consider the following example in Figure \ref{fig:litmus-test}. It is a Store-Buffering program in which the variables X and Y are initialized to 0. Each thread first assigns the value 1 to one variable and then reads the other variable. From the programmer’s perspective, one might expect both a and b to be 1 if the program executes in the order in which the operations are written: Thread 0 writes X and then reads Y, so if Thread 1’s write to Y is already visible, a will be 1; symmetrically, b will be 1 if Thread 0’s write to X is already visible. However, hardware store buffering allows each thread to delay its write and execute the subsequent read before the other thread’s write propagates. In this case, both a and b can be 0: Thread 0 reads Y=0 before Thread 1’s write to Y is visible, and Thread 1 reads X=0 before Thread 0’s write to X is visible. This outcome $a=0, b=0$ is forbidden under Sequential Consistency but is observable on hardware with store buffers, making it the key weak memory behavior of this test. In the execution diagrams that follow, $W(X)=v$ denotes a write of value $v$ to memory location $X$, and $R(Y)=v$ denotes a read from location $Y$ that returns value $v$. Solid arrows represent the global execution order, and dashed arrows represent reads-from (\textit{rf}) edges, connecting each write to the read that observes its value. For instance, we may non-deterministically observe one of the following interleaving orders:

Even though the order of instructions within each thread is preserved according to the program order, threads can interleave arbitrarily. The dashed reads-from (\texttt{rf}) arrows indicate the relationship between write operations and subsequent reads of the same memory location in the interleaving. Depending on the specific interleaving, these read operations may observe values written by different write operations, resulting in distinct assignments to a and b upon termination. The formal study of the exact behaviors concurrent programs can exhibit is a field known as weak memory models.

Memory models are generally categorized into two classes: operational and axiomatic. Operational models are described using buffers and queues and are abstractions derived from hardware behavior \citep{10.1145/1993498.1993520OperationalModel}. In contrast, axiomatic memory models specify disallowed program behaviors, usually by constraining relations on memory accesses \citep{10.1145/2627752}. In this thesis, we use axiomatic memory models.

\subsection{Axiomatic Weak Memory Models}
\label{Background:AWMM}

In Figure \ref{fig:litmus-execution-0-1}, we use lines to show the execution order and write-read pairs. In axiomatic models, these lines represent binary relations, and events represent nodes or instructions. Together, these binary relations compose an execution graph. 

Each event is used to represent the behavior of an instruction. For example, Event R is used to define a read instruction that reads a value from a location, and Event W is used to define a write instruction that writes a value to that location. These addresses are concrete values (e.g., 0x00000000), but for clarity of presentation we denote them using symbols (e.g., x, y) in the examples. These symbolic locations strictly represent distinct locations in memory (i.e., X and Y are strictly different). Each event also has an identifier that is unique within the execution.

\subsubsection{Primitive Relations}

The relations between events are distinguished as primitive relations and derived relations. The latter are entirely defined from the former and are composed of primitive relations. We also define some typical primitive relations~\citep{10.1145/2627752}:

\textbf{Program Order (\texttt{po})}, relating events by their syntactic order: the order among instructions in the program text carries over to execution graphs with \texttt{po} edges. It is a total order per thread. Figure~\ref{fig:litmus-execution-0-1} contains four events. In thread 0, the program order is $e_1 : W(x)=1$ followed by $e_2 : R(Y)=1$. In thread 1, the order is $e_3 : W(Y)=1$ followed by $e_4 : R(X)=1$. We denote these as $\mathit{po}(e_1, e_2)$ and $\mathit{po}(e_3, e_4)$.

\textbf{Reads From (\texttt{rf})}, which relates a W event to the R event reading from it. Every R event reads-from one W event or an initial write event. For instance, when event $e_2 : R(X) = 1$ reads 1 from location $X$, where that value was earlier written by $e_1 : W(X) = 1$, then we have an \texttt{rf} edge from $e_1$ to $e_2$, alternatively denoted as $\mathit{rf}(e_1, e_2)$.

\textbf{Coherence Order (\texttt{co})}, which relates a W event with a W event overwriting it. For instance, when event $e_3 : W(X)=1$ first writes 1 to location $X$, which event $e_4 : W(X)=2$ then overwrites with 2, then we have an edge $\mathit{co}(e_3, e_4)$. Both \texttt{co}-related write events access the same location. Additionally, \texttt{co} is a total strict order per location.

\textbf{Read Modify Write (\texttt{rmw})}, relating the R and W of a successful atomic read-modify-write instruction (RMW). RMW instructions execute atomically, ensuring no other thread can modify the value during the operation. Some variants, like compare-and-swap, can detect if the value has changed before the operation by checking against an expected value. The generated R and W events access the same location and are \texttt{po}-adjacent (i.e., there are no \texttt{po}-intermediate events).

Using these primitive relations, we can represent executions of the Store Buffering litmus test. The execution orders are implicitly determined by these primitive relations. Figure~\ref{fig:litmus-execution-primitive-relation} shows one such execution, where the dotted lines represent the coherence order, the dashed lines represent the reads-from relation, and the thick lines represent the program order. And one such an execution graph is called a \texttt{Candidate Execution}.

A candidate execution consists of a set of events together with a collection of
relations over these events. In general, it is represented as
\[
X = \langle E, \mathit{po}, \mathit{rf}, \mathit{co}, \mathit{rmw} \rangle,
\]
where $E$ is a set of events and $\mathit{po}, \mathit{rf}, \mathit{co}, \mathit{rmw} \subseteq E \times E$ are
relations over $E$. We do not fix the exact set of relations included in a candidate execution.
Additional primitive relations may be introduced later, or some derived
relations may be treated as primitive ones to simplify proofs.


\subsubsection{Derived Relations}

Some relations are composed based on the primitive relations. The following are commonly used derived relations:

\textbf{From-read (\texttt{fr})}: if a read instruction reads from a location that is later overwritten by another write, we relate these two events. For instance, consider $e_1 : r(x) = 1$, $e_2 : w(x) = 1$, and $e_3 : w(x) = 2$. Event $e_1$ first reads the value written by $e_2$, giving $\mathit{rf}(e_2, e_1)$. Later, this value is overwritten by $e_3$, giving $\mathit{co}(e_2, e_3)$. The \texttt{fr} relation is therefore derived as $\mathit{rf}^{-1}\mathbin{;}\mathit{co}$, as illustrated in Figure~\ref{fig:fr-example}. The $^{-1}$ is the inverse of a relation.

For instance, \texttt{rfe} consists of all \texttt{rf} edges between threads; \texttt{coe} denotes external \texttt{co}, and \texttt{fre} denotes external \texttt{fr}. Conversely, \texttt{internal relations} are relations between events within the same thread, such as \texttt{rfi}, \texttt{coi}, and \texttt{fri}.

\subsubsection{Litmus Tests}
\label{sec:litmus-tests}

The Store Buffering example in Figure~\ref{fig:litmus-test} is often used together with a small concurrent program to demonstrate the possible behaviors of a memory model. And in these concurrent programs, the results are always non-deterministic. Such small programs are called litmus tests. We can observe these unexpected behaviors by running a litmus test on real hardware (e.g., RISC-V, ARM, x86, etc.).

The Inria team~\citep{10.1145/2627752} developed a tool called \href{https://diy.inria.fr/}{DIY7} that automatically generates litmus tests for different architectures using their litmus program specifications. For example, the Store Buffering example in Figure~\ref{fig:litmus-test} on x86 can be represented as the following litmus test, \texttt{SB.litmus}:

\begin{lstlisting}[numbers=left, xleftmargin=2em]
X86 SB
"Fre PodWR Fre PodWR"
{ x=0; y=0; }

P0          | P1
MOV [x],$1  | MOV [y],$1
MOV EAX,[y] | MOV EAX,[x]

exists (0:EAX=0 /\ 1:EAX=0)
\end{lstlisting}

A litmus test specification typically consists of three main sections:

\textbf{Initial state (line 3).} This section defines the initial values of registers and memory locations. In the example, both \texttt{x} and \texttt{y} are initialised to \texttt{0}. Initialization to zero may be omitted.

\textbf{Code section (lines 5--7).} This section defines the code that is executed concurrently. In the example, thread \texttt{P0} writes \texttt{1} to \texttt{x} and then reads from \texttt{y} into \texttt{EAX}, while thread \texttt{P1} writes \texttt{1} to \texttt{y} and then reads from \texttt{x} into \texttt{EAX}.

\textbf{Final condition (line 9).} This section specifies constraints on the final values of registers and memory locations. Here, the condition asks whether both threads can read \texttt{0}, i.e., whether \texttt{P0:EAX=0} and \texttt{P1:EAX=0} can hold simultaneously.

The DIY7 toolchains compile such litmus tests into real machine code and collect the results after running them a specific number of times. The semantics of a program is the set of all executions the program can exhibit when running on an architecture \citep{DennisPhdThesis}. 

In the x86 SB example, the \texttt{MOV} instruction can be translated into read or write events. Based on this translation, events and relations between them are generated, while the computations in the program are ignored. The semantics then proposes the possible execution graphs, called candidate executions. We can subsequently use the memory model to determine whether any of the candidate executions are disallowed.

The syntax and usage of litmus tests are not the main focus of this thesis. In \textsc{LeanCat}, we abstract litmus tests as a set of possible \texttt{candidate execution}s directly. Supporting the translation from architecture-specific instructions to candidate executions is time-consuming, and our main task is to compare different memory models. Therefore, we currently provide a \texttt{Candidate Execution} structure as the template.

\subsubsection{Memory Model Semantics}

Having shown how to use binary relations to draw a litmus test, we have not yet demonstrated how to define a memory model. The Inria team~\citep{10.1145/2627752} developed a specification language called Cat, which uses binary relations to define memory models. Many memory models, such as RISC-V, ARM, and X86-TSO~\citep{sewell2010x86tso}, are already defined in Cat; in this section we introduce sequential consistency and total store order as representative examples.

\citet{lam1675439} introduced what is called Sequential consistency: \textit{the result of any execution is the same as if the operations of all the processors were executed in some sequential order, and the operations of each individual processor appear in this sequence in the order specified by its program. A multiprocessor satisfying this condition will be called sequentially consistent.}

The Sequential Consistency (SC) model is the most intuitive memory model for
programmers. It states that the execution of a program must follow the order of
instructions as specified in the program text, while the instructions from
different threads may be interleaved with one another. In other words, all
operations appear to execute in a single global sequence that preserves the
program order of each individual thread.

Let us return to the \textit{Store Buffering} example. Under Sequential
Consistency, the only possible behaviors are those obtained by interleaving the
instructions of the two threads while preserving the program order of each
thread. In this case, there are six possible interleavings:

\begin{enumerate}
\item $W(x)=1 \;\rightarrow\; R(y) \;\rightarrow\; W(y)=1 \;\rightarrow\; R(x)$
\item $W(x)=1 \;\rightarrow\; W(y)=1 \;\rightarrow\; R(y) \;\rightarrow\; R(x)$
\item $W(x)=1 \;\rightarrow\; W(y)=1 \;\rightarrow\; R(x) \;\rightarrow\; R(y)$
\item $W(y)=1 \;\rightarrow\; R(x) \;\rightarrow\; W(x)=1 \;\rightarrow\; R(y)$
\item $W(y)=1 \;\rightarrow\; W(x)=1 \;\rightarrow\; R(y) \;\rightarrow\; R(x)$
\item $W(y)=1 \;\rightarrow\; W(x)=1 \;\rightarrow\; R(x) \;\rightarrow\; R(y)$
\end{enumerate}

These interleavings preserve the program order within each thread while allowing
instructions from different threads to be arbitrarily interleaved. As a result,
every read observes the value written by the most recent write to the same
location in the global execution order.

To formalize Sequential Consistency, memory-model frameworks describe executions
using relations between events. In particular, we define the \textit{communication}
relation, denoted as \texttt{com}. This relation captures how memory operations
interact across threads. It is defined as the union of three relations:

\[
\texttt{let com = rf \;|\; co \;|\; fr}
\]

These relations capture the ways in which operations in different threads
communicate through shared memory. In addition to communication, we consider the \textit{program order} relation, denoted as \texttt{po}. This relation represents the order in which instructions
appear in the program within a single thread and must always be preserved. Sequential Consistency can then be characterized by requiring that the union of
these relations does not contain any cycles:

\[
\texttt{acyclic (po | com)}
\]

The acyclicity condition is important because these binary relations describe
the data-flow between memory events. In particular, read events act as witnesses
for the program behavior, since each read obtains its value from a write event
through the \texttt{rf} relation. This creates a flow of data between events,
and these data-flow relations together constrain the possible ordering of
events, from which a total order can be derived.

However, if a cycle appears in these relations, the ordering constraints become
inconsistent. For example, the \texttt{fr} relation represents that the value
observed by a read is later overwritten by a newer write. If such an \texttt{fr}
edge flows back to a previous event and forms a cycle with program order, it
would imply that a later read in program order observes an older value, which
violates Sequential Consistency. The complete SC model is defined as: 

\begin{figure}[htbp]
\centering
\begin{verbatim}
let com = rf | fr | co
acyclic po | com as sc
\end{verbatim}
\caption{Definition of Sequential Consistency}
\label{fig:sequential-consistency}
\end{figure}

Based on SC, we can build the TSO memory model. The TSO model can be described by a hardware model where each
thread has a write buffer and it's used in X86. A write instruction first stores its value into the buffer instead of writing directly to memory. Because write instructions are
more expensive than reads, the processor can postpone the write and execute
later read instructions first to improve performance. This write buffer architecture is commonly illustrated in the literature \citep{sewell2010x86tso}.

From the programmer's perspective, the program allows the reordering of
write--read pairs within a local thread. That is, a read instruction may happen
before a write event that is \texttt{po}-before the read. Under Sequential
Consistency this behavior is disallowed, so we need to relax the SC model. SC
constrains the program order using the acyclicity condition. To allow this
reordering, we remove write--read pairs from the program order constraint.
Formally, we express this as \texttt{po \textbackslash (W;R)}, where \texttt{;}
denotes relation composition and \texttt{\textbackslash} denotes relation
difference. This removes write--read pairs from the acyclicity constraint,
meaning that read instructions may be reordered before preceding writes in
program order.

In x86, there are additional atomic instructions that are used to avoid the
reordering of write--read pairs, such as \texttt{CMPXCHG}. These instructions
enforce ordering between all preceding and succeeding events, behaving
similarly to a memory fence. Such instructions are represented in the
\texttt{rmw} (read--modify--write) relation. In our previous definitions of
TSO, we did not include these atomic operations, so we need to extend the model
to make TSO-x86 more accurately reflect the behavior of real hardware.

We define \textit{At} (atomic operations) as
\[
\mathit{At} = \mathrm{dom}(\mathit{rmw}) \cup \mathrm{codom}(\mathit{rmw}),
\]
where \texttt{dom} denotes the domain of a relation and \texttt{codom} denotes
the codomain (range) of a relation.

Here, $F$ denotes the set of fence events — special synchronisation instructions that enforce ordering constraints between memory operations on the same thread. Using this set, we define the relation
\[
\mathit{implied} = \mathit{po}\mathbin{;}[\mathit{At} \cup F] \;\cup\; [\mathit{At} \cup F]\mathbin{;}\mathit{po},
\]

And we add this implied to our tso model acyclic constraint, which enforces ordering in the local thread whenever an atomic operation or fence is involved. As a result, the model disallows write--read reordering when an \texttt{rmw} operation occurs between them. This behavior can be observed in
Figure~\ref{fig:tso}. The complete TSO model can be defined as:

\begin{figure}[htbp]
\centering
\begin{verbatim}
let xppo = ((W*W) | (R*W) | (R*R)) & po
let At = domain(rmw) | range(rmw)
let implied = po;[At | F] | [At | F];po
acyclic (implied | xppo | rfe | fr | co) as tso
\end{verbatim}
\caption{Definition of Total Store Order}
\label{fig:tso}
\end{figure}

\subsection{Linux Kernel Memory Model}
\label{Background:LKMM}




We now turn to a more complex memory model, namely the Linux Kernel Memory
Model (LKMM)~\citep{alglave2018frightening}. The Linux system supports a wide range of architectures, such as
ARM, RISC-V, IBM Power, and AMD/Intel. Each architecture has its own memory
model, and therefore they may disagree on the values that a read can return in
a multiprocessor system. Supporting all of them in an efficient and secure way
is challenging. A poorly designed memory model can lead to efficiency problems
and security problems \citep{alglave2018frightening}. The LKMM must therefore be
fairly permissive, in the sense that any behavior allowed by one of these
architectures must also be allowed by the LKMM.  

The most fundamental way to access memory in the Linux kernel is through
\texttt{READ\_ONCE} and \texttt{WRITE\_ONCE}. These primitives prevent the
compiler from optimizing memory accesses by splitting or combining them. For
example, a compiler may combine multiple read operations into a single access
to improve the use of memory bandwidth. The \texttt{ONCE} primitives are used
to prevent such optimizations. In the LKMM, we model the \texttt{READ\_ONCE} instructions as the set
$R[\text{once}]$, and \texttt{WRITE\_ONCE} as the set $W[\text{once}]$.
The brackets denote tags. The notation
$\langle annotatable\text{-}events \rangle[\text{tag}]$
indicates the set of annotatable events that carry the given tag
\citep{10.1145/2627752}.

Besides the READ\_ONCE and WRITE\_ONCE instructions, there are also synchronization primitives in the LKMM. 

\textbf{Acquire and Release}: Full memory barriers disallow any instruction
reordering across the barrier (e.g., mfence in X86), both before and after it. In contrast, an
Acquire operation only prevents later instructions from being reordered before
it, ensuring that all subsequent instructions execute after the Acquire. A
Release operation only prevents earlier instructions from being reordered
after it, ensuring that all prior instructions complete before the Release.
Thus, no instruction before a Release can happen after it, and no instruction
after an Acquire can happen before it. Acquire is associated with a read
instruction, while Release is associated with a write instruction.

\textbf{Fences}: There are four types of fences used in the LKMM. For a read
memory barrier, we have \texttt{smp\_rmb}. For a write memory barrier, we have
\texttt{smp\_wmb}. There is also a full memory barrier, \texttt{smp\_mb}. In
addition, \texttt{smp\_read\_barrier\_depends} is used for architectures that do
not respect dependency ordering.

\textbf{Read-modify-writes}: A read-modify-write (RMW) operation consists of a
read paired with a write. Depending on the primitive, these reads and writes
may be \texttt{ONCE} (for example, \texttt{xchg\_relaxed}), acquire, release, or
surrounded by full fences (for example, \texttt{xchg}).

What makes the LKMM different from the previous models is that it makes
extensive use of tags. These tags are important because they help distinguish
which instruction generated a given event. For example, read events can be
generated by both \texttt{READ\_ONCE} and \texttt{rmw} operations, so we need
tags to differentiate between them. \citet{alglave2018frightening} defines the
following mappings between synchronization primitives and tags:

\begin{figure}[h]
\centering
\begin{center}
\begin{tabular}{ll}
\texttt{READ\_ONCE()} & $R[\text{once}]$ \\
\texttt{WRITE\_ONCE()} & $W[\text{once}]$ \\
\texttt{smp\_load\_acquire()} & $R[\text{acquire}]$ \\
\texttt{smp\_store\_release()} & $W[\text{release}]$ \\
\texttt{smp\_rmb()} & $F[\text{rmb}]$ \\
\texttt{smp\_wmb()} & $F[\text{wmb}]$ \\
\texttt{smp\_mb()} & $F[\text{mb}]$ \\
\texttt{smp\_read\_barrier\_depends()} & $F[\text{rb-dep}]$ \\
\texttt{xchg\_relaxed()} & $R[\text{once}],\; W[\text{once}]$ \\
\texttt{xchg\_acquire()} & $R[\text{acquire}],\; W[\text{once}]$ \\
\texttt{xchg\_release()} & $R[\text{once}],\; W[\text{release}]$ \\
\texttt{xchg()} & $F[\text{mb}],\; R[\text{once}],\; W[\text{once}],\; F[\text{mb}]$
\end{tabular}
\end{center}
\caption{LKMM primitives mapping}
\label{fig:LKMM-events}
\end{figure}

Like we did in TSO and SC, we first need to maintain an order in the local thread. The local orders will be modeled by \textit{ppo} and \textit{SCpv}. The \textit{SCpv} is the sequential consistency per variable, and it states that for each variable the values must follow SC. And \textit{ppo} is preserved program order; this is used to constrain the ordering in the local thread by dependencies and fences. 

The dependencies are composed of address dependencies and data dependencies, denoted $\mathit{addr}$ and $\mathit{data}$ respectively. These dependencies always start with a read event. The data dependencies ($\mathit{data}$) mean that the following instruction will use the result, and the address dependencies ($\mathit{addr}$) mean that the following instruction will use the value of the result as the address. Therefore these dependencies cannot be reordered.

And these dependencies can be categorized into read--write dependencies and read--read dependencies. The read--write dependencies include the \textit{ctrl} dependencies, but the relation has to be the read--write pair as:
\[
\mathit{rwdep} := (\mathit{dep} \cup \mathit{ctrl}) \cap (R \times W)
\]

The read--read dependencies are the address dependencies or the \textit{dep} followed by an \textit{rfi} (starts with a read and ends with a read). The architecture Alpha does not preserve the read--read address dependencies, so LKMM adds the \textit{rd-dep} as the \textit{strong-rrdep} to model Alpha explicitly~\citep{alglave2018frightening}.

\textit{ppo} (preserved program order) is the subset of program order that the hardware must respect — it captures the ordering constraints that cannot be reordered even on a weak memory model. \textit{ppo} uses the dependencies as the \textit{to-r} and \textit{to-w} relations. The \textit{to-w} is modeled as the internal overwrites, and overwrite is from-read (a write overwrites a read) and a coherence order (a write overwrites a write), and together with the \textit{rw-dep} to compose an order of \textit{to-w}. The \textit{to-r} is modeled as \textit{strong-rrdep} or the \textit{rfi-rel-acq}. The \textit{strong-rrdep} is explained above, and the \textit{rfi-rel-acq} is the typical release-write, acquire-read order, and we cannot reorder these release-acquire instructions.

The fence also contributes to the \textit{ppo}. $\mathit{mb}$ denotes the set of full memory-barrier fence events, i.e., $\mathit{mb} = F[\texttt{mb}]$. The \textit{strong-fence} is $\mathit{mb}$; it orders the events separated by \texttt{smp\_mb}. It also orders the events separated by \texttt{rmb} and \texttt{wmb}, or when the first event is the acquire (\textit{po-acquire}) or the second event is the release (\textit{po-rel}). The ppo is defined as: 

\[
\mathit{ppo} := \mathit{rrdep}^{*} \mathbin{;} (\mathit{to\text{-}r} \cup \mathit{to\text{-}w} \cup \mathit{fence})
\]

Besides this local thread ordering (\textit{ppo}), we also need to constrain the order involving other threads. The \textit{prop} (propagation) is used to make sure the ordering of propagation writes. The propagation is used to describe the write instructions interacting with other threads (we ignore the hardware abstraction, e.g., caches and memory systems, because it is complex). That is, when we use a fence related to writes that come from another thread, we need to make sure that all the stores \textit{po}-later than this fence must be executed later than the store (from another thread) to this thread.

They use a value called \textit{A-cumul} to describe this behavior:
\[
\mathit{A\text{-}cumul}(r) := \mathit{rfe}^? \mathbin{;} r
\]

because the \textit{rfe} is the interaction with the store from another thread. We want to make sure the read instruction reads the store propagated from another thread before the fence. Then the \textit{cumul-fence} becomes
\[
\mathit{cumul\text{-}fence} := \mathit{A\text{-}cumul}(\mathit{strong\text{-}fence} \cup \mathit{po\text{-}rel}) \cup \mathit{wmb}
\]

The \textit{strong-fence} and \textit{po-rel} need to be \textit{A-cumul} because they cannot find the correct write from another thread if we do not put the \textit{rfe} on them. Then the \textit{prop} can be defined as
\[
\mathit{prop} := (\mathit{overwrite} \cap \mathit{ext})^? \mathbin{;} \mathit{cumul\text{-}fence}^{*} \mathbin{;} \mathit{rfe}^?
\]

The final definition generalizes the cumulative behaviors of propagation by adding the \textit{overwrites} and \textit{rfe}.

The \textit{ppo} and \textit{prop} compose the \textit{hb} (happens-before) relation. The \textit{ppo} is for the ordering within the same thread, and the \textit{prop} is for the ordering involving other threads. Here, $\mathit{id}$ denotes the identity relation $\Delta$ (every event related to itself), so $\mathit{prop} \setminus \mathit{id}$ restricts \textit{prop} to pairs of distinct events. The \textit{hb} is the union of \textit{ppo} and \textit{rfe} relations, together with the \textit{prop} that relates distinct events. Then we can use \textit{acyclic} to constrain the \textit{hb}, to make sure there is no ordering problem. 

Besides the \textit{hb} (happens-before) relation, they also defined the \textit{pb} (propagates-before) relation. The \textit{pb} relation consists of events that begin with a write originating from another thread (i.e., propagations), followed by a strong fence and then an arbitrary number of \textit{hb} links.
They enforce the acyclicity of \textit{pb} to guarantee that events are overwritten in a manner consistent with the orderings imposed by strong fences.

In summary, the Linux Kernel Memory Model (LKMM) is defined by the following constraints:

\[
\begin{aligned}
&\mathrm{acyclic}(\mathit{po}\text{-}\mathit{loc} \cup \mathit{com}) \\
&\mathrm{empty}(\mathit{rmw} \cap (\mathit{fre} \mathbin{;} \mathit{coe})) \\
&\mathrm{acyclic}(\mathit{hb}) \\
&\mathrm{acyclic}(\mathit{pb})
\end{aligned}
\]

The LKMM is defined in Figure~\ref{fig:lkmm-definitions}. The LKMM also formalizes RCU operations, but these involve very complex and computationally expensive constraints. For this reason, we decided to exclude the RCU-related part from our analysis.
We explain this decision in more detail in Section~\ref{sec:design:cat-subset}.

\begin{figure}[h]
\centering
\begin{align*}
\mathit{dep} &:= \mathit{addr} \cup \mathit{data} \\
\mathit{rwdep} &:= (\mathit{dep} \cup \mathit{ctrl}) \cap (R \times W) \\
\mathit{overwrite} &:= \mathit{co} \cup \mathit{fr} \\
\mathit{to}\text{-}\mathit{w} &:= \mathit{rwdep} \cup (\mathit{overwrite} \cap \mathit{int}) \\
\mathit{rrdep} &:= \mathit{addr} \cup (\mathit{dep} \mathbin{;} \mathit{rfi}) \\
\mathit{strong}\text{-}\mathit{rrdep} &:= \mathit{rrdep}^{+} \cap \mathit{rb}\text{-}\mathit{dep} \\
\mathit{to}\text{-}\mathit{r} &:= \mathit{strong}\text{-}\mathit{rrdep} \cup \mathit{rfi}\text{-}\mathit{rel}\text{-}\mathit{acq} \\

\mathit{fence} &:= \mathit{strong}\text{-}\mathit{fence} \cup \mathit{po}\text{-}\mathit{rel} \cup \mathit{wmb} \cup \mathit{rmb} \cup \mathit{acq}\text{-}\mathit{po} \\
\mathit{ppo} &:= \mathit{rrdep}^{*} \mathbin{;} (\mathit{to}\text{-}\mathit{r} \cup \mathit{to}\text{-}\mathit{w} \cup \mathit{fence}) \\

\mathit{prop} &:= (\mathit{overwrite} \cap \mathit{ext})^{?} \mathbin{;} \mathit{cumul}\text{-}\mathit{fence}^{*} \mathbin{;} \mathit{rfe}^{?} \\
\mathit{hb} &:= ((\mathit{prop} \setminus \mathit{id}) \cap \mathit{int}) \cup \mathit{ppo} \cup \mathit{rfe} \\
\mathit{pb} &:= \mathit{prop} \mathbin{;} \mathit{strong}\text{-}\mathit{fence} \mathbin{;} \mathit{hb}^{*}
\end{align*}

\subsection{BPF Memory Model}
\label{Background:BPF}

BPF (Berkeley Packet Filter)~\citep{mccanne1993bpf} is a technology that allows programs to run safely inside the Linux kernel~\citep{linux-kernel} without modifying kernel source code or loading kernel modules. It was originally designed for network packet filtering, but its extended version, eBPF, has since become widely used in industry for performance monitoring, tracing, and security enforcement. Because eBPF programs execute within the kernel, they interact directly with the kernel's memory subsystem and are subject to the same weak memory ordering behaviors as the rest of the kernel.

Despite this close relationship, no formal memory model for BPF existed prior to 2024. The BPF memory model was introduced in 2024 through a proposal discussed on \href{https://lwn.net/Articles/976071/}{LWN.net}~\citep{bpf-memory-model-lwn}. That proposal identifies two principal sources of weak memory behaviors in BPF: atomic instructions (such as \texttt{xchg} and \texttt{cmpxchg}), and jump instructions. Understanding these behaviors is important because they represent the bare minimum of ordering guarantees that most BPF programs currently rely on. In the absence of a formalized model, it is impossible to determine whether a bug caused by reordered memory operations is a fault in the BPF program, the compiler, or the BPF just-in-time (JIT) compiler.

The BPF memory model follows the same design philosophy as the LKMM: it should be as weak as possible, avoiding prohibiting behaviors that the supported architectures permit. However, BPF instructions cannot be directly mapped to the LKMM, because the LKMM is defined at the level of C-language primitives, whereas BPF is closer to assembly. Consequently, some constraints required by BPF semantics fall outside what the LKMM can express. The designers therefore chose to adapt the model from both the LKMM and the AArch64 (ARMv8) memory model, as AArch64 provides a more expressive foundation for instruction-level atomic operations. In particular, the BPF memory model adopts the atomic memory operation (AMO) design from AArch64, together with acquire (\texttt{AQ}) and release (\texttt{RL}) annotations. The BPF memory model is defined in \href{https://github.com/herd/herdtools7/blob/master/herd/libdir/bpf.cat}{herd/libdir/bpf.cat}.

The BPF model exhibits the same high-level structure as the LKMM: it enforces coherence (\textit{Coherence}), happens-before (\textit{Happens-before}), propagation (\textit{Propagation}), and atomicity (\textit{Atomic}) constraints. The key difference lies in how synchronization is expressed. Rather than relying on Linux kernel fence primitives declared in a bell file, the BPF model uses AArch64-style atomic memory operations as its synchronization backbone. The \textit{ppo} relation is built from explicit synchronization via \texttt{po\_amo\_fetch} and release-consistency (\texttt{rcpc}), together with syntactic dependencies (address, data, and control) and pipeline dependencies. The \textit{prop} relation captures cumulative propagation in the same spirit as the LKMM, using \textit{A-cumul} to account for stores arriving from other threads before a synchronizing operation. In this way, the BPF memory model inherits the structural discipline of the LKMM while grounding its synchronization primitives in the AArch64 memory model.



\section{Lean 4 and Its Metaprogramming Framework}
This section introduces the necessary background for readers unfamiliar with Lean 4; readers already familiar with it may skip ahead to Section~\ref{chapter:design}. We cover only the most important concepts used in this project and do not introduce complex topics such as monads. Before presenting the key features of Lean 4, we first explain why we chose it as our tool for this project.

\subsection{Lean 4}

Lean 4 is an interactive theorem prover (ITP). Traditionally, most proofs for memory models have been carried out on paper, for example in \citep{podkopaev2019bridging, 10.1145/2627752, batty2011mathematizing, doko2016program}. Although these works make important theoretical contributions, they are prone to subtle issues, such as undetected errors in existing models \citep{chakraborty2016fixing, chakraborty2017grounding} or the need for later repairs \citep{10.1145/3140587.3062352Reparing}. By using an ITP, we can dramatically increase the reliability of our proofs: they are mechanically checked by a computer, greatly reducing the chance of missed edge cases.

Lean 4~\citep{deMoura2021lean4} is not only just an ITP, it is also a functional programming language like Haskell and OCaml, which means it is a general-purpose programming language that supports computation and IO. This gives us the ability to read and parse the Cat files directly from storage, instead of manually migrating the Cat files into Lean~4.

For readers unfamiliar with interactive theorem provers, the following simple example illustrates how they work. We can experiment with more examples in the \href{https://adam.math.hhu.de/#/g/leanprover-community/nng4/}{Lean 4 Natural Number Game}. Suppose we want to prove that for all natural numbers \(n\), \(n + 0 = n\). While this equality is obvious at a glance, in a formal system we must justify it using previously proven theorems. In Lean 4, this is achieved using commands called \emph{tactics}. One particularly useful tactic is \texttt{rw} (rewrite). There exists a theorem \texttt{add\_zero} stating that \(\forall a, a + 0 = a\). If our goal is to prove \(n + 0 = n\), we can apply the tactic \texttt{rw [add\_zero]}. This replaces \(n + 0\) with \(n\), immediately closing the goal.


More generally, the \texttt{rw} tactic allows us to replace any subexpression that matches the left-hand side of a known equality with the corresponding right-hand side. Multiple rewrites with \emph{different} theorems can be chained together. For example, to prove \((n + 0) \times 1 = n\), we first apply \texttt{rw [add\_zero]}, which rewrites \(n + 0\) to \(n\), reducing the goal to \(n \times 1 = n\). We then apply \texttt{rw [mul\_one]}, which uses the theorem \(\forall a,\, a \times 1 = a\) to rewrite \(n \times 1\) to \(n\), reducing the goal to the trivial equality \(n = n\), which is closed automatically by \texttt{rfl}. This example reflects a fundamental principle in Lean 4: even the simplest intuitive facts must be formally justified by applying previously established theorems. Tactics provide a systematic and interactive way to do so. While this particular example may appear trivial, we will explore more powerful tactics and theorems later.

\subsection{Automated Proof Search with Aesop}
\label{subsec:aesop}

To mitigate the practical overhead of manual proof construction, we incorporate the \texttt{aesop} (Automated Extensible Search for Omniscient Proofs) tactic ~\citep{ayers2023aesop} into the \texttt{LeanCat} framework. As discussed in Section 4.4, establishing acyclicity and monotonicity often requires significant "tedious" effort in chaining relational properties and constructing transitive paths. \texttt{aesop} is an extensible proof search tactic for Lean 4, broadly similar in function to Isabelle's \texttt{auto}, designed to bridge this gap by performing a rule-based, best-first search to close proof goals.

The tactic operates by maintaining a search tree that alternates between goal nodes and rule application (\textit{rapp}) nodes. This tree is explored based on a collection of user-defined rules, which are categorized into distinct phases. In the context of \texttt{LeanCat}, we leverage several rule builders—such as \texttt{apply}, \texttt{forward}, and \texttt{cases}—to automate structural reasoning. For example, by registering the constructors of the \texttt{NonEmpty} predicate or the cases for \texttt{MyList} as rules, we can automate the "tricky" task of creating \texttt{TransGen} chains during cycle detection. 

Crucially, \texttt{aesop} remains a "tactic proposer" that operates outside the trusted core. While it can be called as \texttt{aesop?} to generate a human-readable tactic script and avoid performance penalties, every proof it generates must ultimately be elaborated into a proof term accepted by the Lean 4 kernel. This ensures that while we automate the mechanical aspects of proof engineering, the logical integrity of the memory model verification remains strictly checked by the ITP.

\subsection{Metaprogramming in Lean 4}

Another important reason for choosing Lean 4 is its extensibility. Lean 4 provides a metaprogramming framework that allows users to define domain-specific languages (DSLs) directly within the system. Informally, metaprogramming refers to writing programs that manipulate or generate other programs. In conventional programming, programs must follow a predefined syntax understood by the compiler or interpreter. The process of parsing this syntax and translating it into intermediate representations, such as abstract syntax trees (ASTs) or control-flow graphs (CFGs), belongs to the meta-level, whereas writing ordinary programs belongs to the object-level.

In most systems, meta-level functionality is implemented in a different language from the object-level language. However, Lean 4 integrates both levels within a single framework: most meta-level code is written in Lean itself. This design enables users to extend the language by defining new syntax constructs and providing corresponding elaboration procedures that translate them into core Lean expressions.

This capability is often described as reflection, where meta-level functionality is made directly accessible at the object-level. As a result, users can introduce custom notation or commands and use them immediately within the same development environment. For example, new commands can be defined concisely using elab declarations.

In practice, Lean 4 metaprogramming frequently involves working directly with its internal representations of programs. In particular, the concrete syntax tree (CST), represented by the Syntax type, captures the parsed structure of code, while the abstract syntax tree (AST), represented by the Expr type, corresponds to its elaborated form. These representations are especially important when implementing tactics and other advanced features. Consequently, although Lean 4 provides powerful metaprogramming capabilities, mastering them requires a solid understanding of these underlying structures.

Lean~4 metaprogramming provides two ways to operate on the grammar: macros and elaborators. A macro is the simpler option — it is a function (with side effects) that converts one syntax term into another, similar to macros in Rust. An elaborator is more powerful: it operates at the CST level, converts terms into AST form, and can also manipulate the environment (namespaces, scopes, etc.). In this project we use macros as the primary mechanism because they offer a direct and straightforward mapping; elaborators are used for the more complex cases that involve environment manipulation.

To illustrate, suppose we want to translate the notation let r = co | po into the Lean 4 definition def r := co.union po, where co and po are binary relations. We can start by defining a syntax rule for the Lean 4 parser like this: 

syntax "let" cat\_ident "=" expr : inst


This looks similar to a regular expression pattern. The \verb|syntax| keyword declares a new piece of concrete syntax, and \verb|: inst| specifies that this syntax belongs to the syntax category \verb|inst| (a node kind in the concrete syntax tree used to organize and group related syntax constructs), we can use \texttt{syntax\_cat} to define a syntax category in Lean 4. Here, \verb|cat_ident| stands for the identifier (e.g., \verb|r|), and \verb|expr| represents the right-hand side. In our case, the binary operation co | po. Next, we need to define the \verb||| operator itself for expressions with the following rule: 

syntax:51 expr:51 "|" expr:50 : expr

The natural number after \verb|syntax| and after each \verb|expr| specifies precedence levels. This declaration means: The entire production (the binary | operator) has precedence 51.
The left operand is parsed with precedence 51 (same as the operator).
The right operand is parsed with precedence 50 (strictly lower).
This asymmetry encodes right-associativity and follows the Pratt parsing technique~\citep{pratt1973top}.

In Lean 4, a higher number indicates higher (stronger/tighter) precedence. Therefore, for an expression such as r1 | r2 | r3, the parser groups it as ((r1 | r2) | r3), that is left-associative. This happens because the right operand is restricted to precedence $> 50$. If we tried to group it the other way, as (r1 | (r2 | r3)) without parentheses, and the subexpression (r2 | r3) would need to be parsed in a context expecting precedence $> 50$, but the inner | operator has precedence 51, which conflicts with the allowed range. Hence, such a grouping is naturally rejected by the parser, enforcing left associativity.

After we have defined how to parse the \texttt{inst} category (and the associated syntax), the next step is to assign semantics to it. Lean~4 provides multiple ways to handle syntax transformations and elaboration, summarised in Figure~\ref{fig:meta-api}. According to the Lean~4 metaprogramming documentation~\citep{lean4metaprog}, the pipeline follows the order \texttt{syntax} $\to$ \texttt{macro} $\to$ \texttt{elab}: after parsing, any applicable macros are applied iteratively, and finally a single elaborator converts the resulting syntax into a typed expression.

The mechanisms fall into two broad groups. \emph{Macros} (\texttt{notation}, \texttt{macro}, \texttt{macro\_rules}) are \texttt{Syntax~→~MacroM~Syntax} functions: they transform one piece of syntax into another. \texttt{notation} and \texttt{macro} are high-level conveniences that automatically declare both a \texttt{syntax} rule and the corresponding macro; \texttt{macro\_rules} provides explicit pattern matching over syntax while handling boilerplate automatically. Macros are most suitable for defining syntax sugars, notations, and shortcuts~\citep{lean4metaprog}; as soon as types or control flow are involved, a macro is no longer the right tool.

\emph{Elaborators} (\texttt{elab}, \texttt{elab\_rules}) are responsible for translating user-facing syntax into typed expressions that the rest of the compiler can process~\citep{lean4metaprog}. They run inside \texttt{TermElabM} or \texttt{CommandElabM}, giving them full access to the type environment, expected types, and metavariables. In this project, we make use of all these mechanisms, but we will not introduce them all in this section. Instead, we focus on illustrating how \texttt{macro\_rules} works through the following example.

As mentioned earlier, a macro is essentially a conversion from one piece of syntax to another. In Lean~4, \texttt{antiquotations} (brackets prefixed with `) are used to capture and splice parts of the input syntax into the output syntax. These act like placeholders in a pattern-matching regular expression style template for syntax trees. Here is how we can implement the desired translation using \texttt{macro\_rules}:

\begin{verbatim}
macro_rules
  | `(let $nm:cat_ident = $e) => `(abbrev $nm := $e)
\end{verbatim}

This rule matches the pattern \texttt{let <identifier> = <expression>} (where \$nm captures the identifier from the \texttt{cat\_ident} category, and \$e captures the right-hand side expression) and rewrites it to an \texttt{abbrev} declaration (a lightweight definition in Lean).

For the binary | operator (which we want to map to set union), we add another rule:

\begin{verbatim}
macro_rules
  | `($e₁:expr "|" $e₂:expr) => `(Set.union $e₁ $e₂)
\end{verbatim}

This is a simple illustration of how the meta-programming framework works. In our project, the implementation is more complex and it is not necessary to introduce all of them. After we defining the Macro rules, Lean 4 checks that the final elaborated terms are type-safe using its small trusted kernel. Macros and elaborators operate outside the kernel and can define arbitrary syntax or transformations, but their output must ultimately be accepted by the kernel~\citep{deMoura2021lean4}. This means even if we define unsafe or incorrect syntax transformations, they cannot break soundness because the kernel will reject any ill-typed terms.
